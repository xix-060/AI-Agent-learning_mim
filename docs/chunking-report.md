# 切块策略实验报告

## 📊 实验结果汇总

| 配置 | 切块数 | 平均块长 | 检索准确率 |
|------|:------:|:--------:|:----------:|
| fixed-200 | 6 | 193 字符 | ✅ 100% (5/5) |
| fixed-500 | 3 | 386 字符 | ✅ 100% (5/5) |
| fixed-1000 | 2 | 579 字符 | ✅ 100% (5/5) |
| sentence-3 | 8 | 130 字符 | ⚠️ 80% (4/5) |
| sentence-5 | 5 | 209 字符 | ⚠️ 80% (4/5) |
| paragraph | 8 | 130 字符 | ✅ 100% (5/5) |

---

## 1️⃣ 哪种切块策略准确率最高？

**本次实验中，有 4 种策略达到了 100% 的检索准确率：**

- **固定窗口**：fixed-200, fixed-500, fixed-1000 ✅
- **按段落**：paragraph ✅

而 **按句子切分**（sentence-3, sentence-5）仅达到 **80%**，有 1 个问题未检索到正确内容。

### 为什么句子切分表现较差？

按句子切分可能会**切断跨句依赖的信息**。例如，某个关键概念可能在第 1 句引入，在第 2 句才给出详细定义。如果只取 3-5 句为一个块，可能无法完整捕获这种上下文关联。

```
示例：Transformer 的描述
  第 1 句："Transformer 模型于 2017 年提出"
  第 2 句："它完全基于自注意力机制"
  第 3 句："摒弃了传统的循环结构"

sentence-3 能完整保留上下文 ✅
但如果定义分散在 6 句中，sentence-3 就会丢失信息 ❌
```

---

## 2️⃣ 块太小（200字符）和块太大（1000字符）各有什么问题？

### 📉 块太小的问题（fixed-200）

虽然本次实验 fixed-200 达到了 100%，但这**不代表它是完美的**：

| 问题 | 影响 | 实验数据 |
|------|------|---------|
| **索引膨胀** | 文档被切成 6 块，是 fixed-1000 的 3 倍 | 6 块 vs 2 块 |
| **检索开销** | 需要检索、排序更多块，延迟增加 | 每请求处理更多候选 |
| **上下文截断** | 可能在句子中间切断，破坏语义完整性 | 平均 193 字符/块 |

### 📈 块太大的问题（fixed-1000）

本次 fixed-1000 也达到了 100%，但同样存在隐患：

| 问题 | 影响 | 实验数据 |
|------|------|---------|
| **信息稀释** | 大块包含无关内容，embedding 精确度下降 | 平均 579 字符/块 |
| **冗余填充** | 不相关的段落被引入 LLM，浪费 token | 每块包含更多噪声 |
| **Top-K 限制** | 如果关键信息被稀释，可能排不到 Top-K | 仅 2 块可检索 |

### ⚖️ 本次实验为什么块大小影响不明显？

**样本量太小！** 本次测试仅用了 5 个问题，且所有关键信息都在相对集中的位置。在**生产环境**中（长文档、多主题），块大小的影响会显著放大：

```
生产场景预期表现：
  fixed-200：可能因上下文不完整导致答案错误 ❌
  fixed-1000：可能因噪声干扰导致生成质量下降 ❌
  fixed-500：平衡点 ✅
```

## 3️⃣ 句子切分 vs 固定窗口，哪个更好？

| 维度 | 固定窗口 | 句子切分 |
|------|---------|---------|
| **准确率** | 100% ✅ | 80% ⚠️ |
| **语义完整性** | 可能切断句子 ❌ | 保留完整句子 ✅ |
| **块大小控制** | 精确可控 ✅ | 依赖句长分布 ❌ |
| **边界对齐** | 需额外处理 overlap | 自然对齐 ✅ |
| **实现复杂度** | 低 ✅ | 需分句逻辑 ❌ |

### 🎯 结论：固定窗口 > 句子切分

**本次实验数据明确显示固定窗口更优**，原因：

1. **准确率更高**：固定窗口 100% vs 句子切分 80%
2. **可控性更强**：可以根据 embedding 模型的 token 限制调整块大小
3. **overlap 机制**：通过 10-20% 的重叠率弥补"切断"问题

### 💡 例外情况

如果文档有**良好的段落结构**（如技术文档、论文），**按段落切分**可能是最优选择：
- paragraph 策略达到了 100% 准确率
- 保留了自然的语义边界
- 无需复杂的 overlap 计算

## 4️⃣ 我的推荐参数

### 🏆 首选：fixed-500（固定窗口 500 字符）

```json
{
  "method": "fixed",
  "chunk_size": 500,
  "overlap": 50,
  "reason": "最佳平衡点"
}
```

**推荐理由：**

| 指标 | fixed-200 | fixed-500 | fixed-1000 |
|------|-----------|-----------|------------|
| 准确率 | 100% | **100%** | 100% |
| 块数 | 6（膨胀） | **3（适中）** | 2（过少） |
| 平均块长 | 193 | **386** | 579 |
| Token 估算 | ~97 tokens | **~193 tokens** | ~290 tokens |

**为什么不是 200 或 1000？**

- **不用 200**：块太碎，索引开销大，无法覆盖完整上下文
- **不用 1000**：块太大，embedding 稀释，且超出部分 embedding 模型的最佳窗口

### 🥈 备选：按段落切分

```json
{
  "method": "paragraph",
  "reason": "当文档有良好段落结构时"
}
```

**适用场景：**
- 技术文档、API 文档
- Markdown 文件（标题+段落结构清晰）
- 学术论文

### ⚠️ 注意事项

本次实验的**样本数据仅 1157 字符**（约 5-6 个段落），对于真实场景（数千到数万字符文档），建议：

1. **扩大测试集**：至少 50+ 个问题
2. **多种文档类型**：技术文档、新闻、书籍、代码
3. **A/B 测试**：在真实问答场景中对比不同参数
