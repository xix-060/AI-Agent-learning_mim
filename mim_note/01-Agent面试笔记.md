## 1. Agent 的本质是什么

### 一句话破题

Agent 是一个能**自主感知→规划→行动→再感知**形成闭环的 AI 系统，和普通大模型的本质区别就两个字：**自主**。

### 核心三问

1. Agent 和"带插件的大模型"有什么区别？
2. 普通大模型有哪三大局限，Agent 怎么逐个突破的？
3. 为什么 Agent 概念很早就有，现在才爆发？

### 类比记忆法

把普通大模型想象成一个**坐在椅子上博学但手脚被绑住的教授**——你问他问题他能答，但他不能站起来去书架翻书、不能打电话、也不记得昨天你问过他什么。

Agent 就是给这个教授**松了绑**：给了腿（工具调用）、给了笔记本（记忆）、给了行程表（规划），然后说"这个项目交给你了，自己搞定"。

| 普通 LLM 的局限 | Agent 的解法 | 类比 |
|-----------------|-------------|------|
| 知识冻结在训练截止日 | 接搜索引擎等工具 | 给了腿，能去查最新资料 |
| 只能输出文字不能行动 | 工具调用执行真实操作 | 给了手，能发邮件、写文件 |
| 每次对话失忆 | 短期+长期记忆系统 | 给了笔记本，记得住上下文 |

### 深度拆解

**Agent 的运作闭环：**

```
感知（收到目标）→ 规划（拆解成步骤）→ 行动（调用工具执行）→ 再感知（拿到结果反馈）→ 循环...
```

这个循环里最容易被忽略的是**"再感知"**。很多人理解 Agent 就停在"能调工具"，但真正让 Agent 成为 Agent 的，是它拿到工具返回结果后会**重新判断**：

- 结果对不对？
- 和预期一致吗？
- 下一步要调整吗？

这种"边做边看边调整"的能力，才是自主性的体现。

**爆发的三个条件：**

| 条件 | 具体表现 |
|------|---------|
| 模型能力跨过门槛 | GPT-4、Claude 3 的推理能力能支撑多步决策 |
| 工具调用标准化 | Function Calling 让模型输出结构化 JSON，不再靠解析自然语言 |
| 配套生态成熟 | LangChain/LlamaIndex 降门槛，向量数据库解决记忆存储 |

### 面试话术

> "我理解的 Agent，本质上是一个能自主完成目标的 AI 系统。和普通大模型最核心的区别在于'自主闭环'——你给它一个复杂目标，它自己拆成多步，通过工具调用跟外部世界交互，每一步的结果反馈回来指导下一步，直到任务完成。
>
> 普通大模型有三个局限：知识冻结、不能行动、没有持续状态。Agent 分别用工具调用、工具执行、记忆系统来突破。但最关键的不是这些零件，而是'感知→规划→行动→再感知'这个闭环——某一步失败了它会换一种方式重试，不是一条路走到黑。
>
> Agent 概念很早就有，现在才爆发是因为三个条件同时成熟了：模型推理能力够了、Function Calling 标准化了工具调用、LangChain 这些框架降低了开发门槛。"

### 雷区雷达

| 坑 | 正确理解 |
|----|---------|
| 把 Agent 等同于"带插件的大模型" | 插件/工具调用只是 Agent 的一部分，关键是自主规划和闭环 |
| 说"模型自己执行工具" | 模型只做决策输出 JSON，真正执行的是你的代码 |
| 忽略"再感知"环节 | 没有结果反馈和调整，就只是自动化脚本不是 Agent |

## 2. Agent 的四大核心组件

### 一句话破题

一个完整的 Agent = **大脑（LLM）+ 手脚（工具系统）+ 记性（记忆系统）+ 谋略（规划模块）**，缺任何一个都跑不了复杂任务。

### 核心三问

1. 四个组件分别干什么？哪个最重要？
2. 工具系统里"决策与执行分离"是什么意思？
3. 记忆系统的短期和长期有什么本质区别？

### 类比记忆法

把 Agent 想象成一家**一人公司**：

| 组件 | 公司里的角色 | 干什么 |
|------|------------|--------|
| LLM | 老板 | 所有决策经过它拍板，判断下一步做什么 |
| 工具系统 | 外包执行团队 | 老板说"去查竞品数据"，团队真正去查 |
| 记忆系统 | 公司档案室 | 存历史合同、客户偏好、项目经验 |
| 规划模块 | 项目经理 | 把"做个竞品分析"拆成搜索→整理→对比→写报告 |

### 深度拆解

**LLM（大脑）**

- System Prompt 是 Agent 的"人格设定"，定义角色、行为边界、输出格式
- 模型选择要看三个因素：推理能力（决定能否正确拆任务）、工具调用稳定性（JSON 格式不出错）、上下文窗口大小（复杂任务跑十几步很容易撑满）
- 工程实践：大模型做核心决策，小模型做简单任务（意图分类、格式提取），搭配使用降成本

**工具系统（手脚）**

核心设计原则：**决策与执行分离**。

```python
# 工具定义里没有一行执行逻辑，只有"说明书"
tools = [{
    "name": "get_weather",
    "description": "查询指定城市的实时天气",  # 这个字段最关键
    "parameters": {"city": {"type": "string"}}
}]
# 模型读了说明书 → 决定调哪个、参数填什么 → 输出 JSON
# 你的代码拿到 JSON → 真正执行 → 结果塞回对话
```

description 是模型判断"该不该用这个工具"的唯一依据。写得含糊（比如只写"查询数据"），模型就会瞎猜。

**记忆系统（记性）**

| 类型 | 存在哪 | 存什么 | 活多久 |
|------|--------|--------|--------|
| 短期记忆 | context window | 当前任务的中间状态 | 任务结束就清 |
| 长期记忆 | 向量数据库 | 跨任务的用户偏好、历史经验 | 持久保存 |

长期记忆可细分：语义记忆（事实知识）、情景记忆（具体经历）、程序记忆（操作流程）。

**规划模块（谋略）**

简单任务一步搞定，复杂任务需要拆解。底层依赖 LLM 推理能力，技术手段包括 CoT（思维链）和 ToT（思维树）。

四组件协作的节奏：**规划→决策→执行→结果存记忆→再决策**，循环往复。

### 面试话术

> "一个完整的 Agent 系统由四个核心组件构成：LLM 是大脑负责决策，工具系统是手脚负责执行，记忆系统保持状态连贯性，规划模块把复杂目标拆成可执行步骤。
>
> 其中我认为最容易在面试中被追问的是工具系统的'决策与执行分离'——工具定义里只有说明书没有执行逻辑，模型负责决定调什么工具、参数填什么，以 JSON 格式输出，你的代码负责真正执行。模型始终只是大脑，不是手脚。
>
> 记忆系统要分清短期和长期：短期记忆在 context window 里，存当前任务的中间状态，任务结束就清空；长期记忆用向量数据库，跨任务保存用户偏好和历史经验，通过语义检索调用。"

### 雷区雷达

| 坑 | 正确理解 |
|----|---------|
| 只说 LLM 和工具两个组件 | 记忆和规划是让 Agent 能跑复杂任务的关键 |
| 说"记忆就是上下文" | 短期在 context window，长期在向量数据库，机制完全不同 |
| 工具描述随便写 | description 是模型选工具的唯一依据，需要反复调优 |

## 3. Workflow、Agent、Tools 三者什么关系

### 一句话破题

Tools 是零件，Workflow 是固定流水线，Agent 是会自己决定走哪条线的灵活工人——**控制权的归属**是三者的分界线。

### 类比记忆法

| 概念 | 类比 | 控制权 |
|------|------|--------|
| Tools | 锤子、螺丝刀 | 没有控制权，是被使用的对象 |
| Workflow | 流水线作业指导书 | 控制权在开发者手里，路径写死了 |
| Agent | 经验丰富的维修工 | 控制权在 Agent 手里，自己判断用哪个工具、按什么顺序 |

### 深度拆解

```
工具（Tools）：单一能力单元，被调用方
    ↓ 被组合进
工作流（Workflow）：预定义的执行路径，if/else 驱动
    ↓ 被自主选择
Agent：动态决策执行路径，LLM 驱动
```

Workflow 的路径是**写死的**：第1步做A，第2步做B，第3步做C。即使中间结果出乎意料，也只会走预设的分支。

Agent 的路径是**动态的**：每一步做完，LLM 根据结果重新决定下一步干什么。没有预设路径，完全看当前情况。

### 面试话术

> "这三个概念是嵌套关系。Tools 是最底层的能力单元——搜索引擎、数据库查询、代码执行器，它们本身没有智能。Workflow 是把多个 Tools 按预设路径串联起来，控制权在开发者手里，路径是写死的。Agent 则把控制权交给 LLM，由模型根据每一步的结果动态决定下一步做什么、用哪个工具。
>
> 实际项目中不是非此即彼，通常是混合使用：确定性强的部分用 Workflow 保证稳定，需要灵活判断的部分用 Agent。"

## 4. Agent 设计范式全景

### 一句话破题

设计范式就是"Agent 怎么思考和行动的套路"——从简单的顺序执行，到边想边做的 ReAct，再到先规划后执行的 Plan-and-Execute，复杂度递增。

### 深度拆解

| 范式 | 核心思路 | 适合场景 | 缺点 |
|------|---------|---------|------|
| 顺序执行 | 按固定步骤一步步做 | 简单确定性任务 | 不灵活 |
| ReAct | 思考→行动→观察循环 | 探索性任务、信息检索 | 容易跑偏 |
| Plan-and-Execute | 先出完整计划再执行 | 目标明确的多步骤任务 | 计划可能不适应变化 |
| Reflection | 做完一步回头检查 | 需要高质量输出的场景 | 增加 token 消耗 |
| 混合模式 | 规划+ReAct+反思组合 | 复杂生产场景 | 实现复杂度高 |

## 5. ReAct 推理模式

### 一句话破题

ReAct = **Reasoning + Acting**，让 LLM 在推理链里插入真实的工具调用，"想一步做一步看一步"，推理给行动方向，行动给推理事实。

### 核心三问

1. ReAct 和 CoT 的本质区别是什么？
2. ReAct 的循环是谁在驱动——模型还是代码？
3. ReAct 有什么致命缺陷？

### 类比记忆法

CoT 是**闭卷考试中的草稿纸**——你把推导过程写下来，但没法翻书查资料。

ReAct 是**开卷考试**——写到一半不确定就去翻书（调工具），翻到答案继续写。

纯 Act-only（只行动不思考）是**无脑抄书**——不停翻书但不思考翻到的东西对不对，抄错了一路错下去。

### 深度拆解

**核心循环：**

```
Thought（思考）→ Action（行动）→ Observation（观察）→ 循环... → Final Answer
```

**具体示例：**

```
Thought: 需要两家公司的市值数据，先查苹果
Action: search("苹果公司 2024 年市值")
Observation: 约 3.5 万亿美元

Thought: 苹果数据有了，再查谷歌
Action: search("谷歌 2024 年市值")
Observation: 约 2.1 万亿美元

Thought: 两个数据都有了，3.5 > 2.1，苹果更高
Final Answer: 苹果市值约3.5万亿，谷歌约2.1万亿，苹果更高
```

**关键认知：循环是代码驱动的，不是模型自己转的。**

```python
def react_agent(question, tools, max_steps=10):
    history = []
    for _ in range(max_steps):
        # 每次把完整历史传给模型，让它输出下一步
        response = llm.generate(prompt + "\n".join(history))
        if "Final Answer" in response:
            return response  # 任务完成
        action, args = parse_action(response)  # 解析行动
        result = tools[action](args)  # 代码执行工具
        history.append(response)
        history.append(f"Observation: {result}")  # 结果填回历史
```

模型每次只做一件事：根据历史输出 Thought + Action。解析、执行、拼回历史，全是代码的事。

**两个致命缺陷：**

| 缺陷 | 说明 | 类比 |
|------|------|------|
| 循环漂移 | 没有全局计划约束，跑着跑着偏离目标 | 没导航的旅行者，被路边风景吸引越走越远 |
| 错误传播 | 中间某步出错，后面全废，没有回头检查机制 | 写代码不测试，一行错了后面全跟着错 |

### 面试话术

> "ReAct 是 Reasoning 和 Acting 的结合，核心循环是'思考→行动→观察'。它解决了 CoT 只能纯文字推理、拿不到实时数据的局限——推理给行动提供方向，行动为推理提供事实反馈。
>
> 需要特别强调的是，这个循环不是模型自己在转，是代码框架在驱动。模型每次只输出 Thought + Action，代码负责解析、执行工具、把 Observation 填回历史、再传给模型进入下一轮。
>
> ReAct 有两个实战局限：一是循环漂移，没有全局计划约束容易偏离目标；二是错误传播，中间某步出错后面全废，没有内置反思机制。这也是为什么实际项目里经常用 Plan-and-Execute 做全局规划，ReAct 做每步执行。"

## 6. ReAct、Plan-and-Execute、Reflection 怎么选

### 一句话破题

ReAct 是"边走边问路"，Plan-and-Execute 是"先看地图再出发"，Reflection 是"走完一段回头看看对不对"——**实际项目里三者组合用**。

### 深度拆解

| 维度 | ReAct | Plan-and-Execute | Reflection |
|------|-------|-----------------|------------|
| 核心思路 | 每步根据当前结果决策 | 先出完整计划再按计划执行 | 做完一步/整个任务后审视质量 |
| 优势 | 灵活，能随时调整 | 有全局视野，不容易跑偏 | 能发现并纠正错误 |
| 劣势 | 容易漂移 | 计划可能不适应变化 | 增加额外 LLM 调用成本 |
| 适合 | 探索性信息获取 | 目标明确的多步骤任务 | 需要高质量输出的场景 |

**工程最佳实践：混合模式**

```
Plan-and-Execute 做全局规划（大模型）
    ↓ 每一步用
ReAct 做灵活执行（小模型，省成本）
    ↓ 关键步骤加
Reflection 做质量检查
```

大小模型搭配能降低 70%~90% 的 LLM 调用成本。

### 面试话术

> "这三种范式不是互斥的。ReAct 灵活但容易跑偏，Plan-and-Execute 有全局视野但计划可能僵化，Reflection 能纠错但有额外成本。
>
> 实际项目中我倾向于混合使用：用 Plan-and-Execute 做全局规划确定大方向，每一步用 ReAct 做灵活执行，关键步骤加 Reflection 做质量检查。规划阶段用大模型保证计划质量，执行阶段用小模型省成本，这种搭配能降 70%~90% 的调用成本。"

## 7. 复杂任务怎么拆

### 一句话破题

任务拆分的本质是**降低单步复杂度**——把一个 LLM 一次搞不定的大任务，变成多个 LLM 能稳定搞定的小任务。

### 类比记忆法

让你一个人一口气写完一篇 50 页的毕业论文，你会崩溃。但如果说"先写提纲→再写文献综述→再写方法→再写实验→最后整合"，每步都可控，就能搞定。

Agent 的任务拆分也是这个逻辑。

### 深度拆解

**为什么要拆：**
- LLM 上下文窗口有限，复杂任务信息量太大放不下
- 单步任务越复杂，模型出错的概率越高
- 拆分后每步可以并行执行，提高效率

**怎么拆：**

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| 按阶段拆 | 把任务分成"调研→分析→输出"等阶段 | 有明确流程的任务 |
| 按依赖拆 | 找出步骤间的依赖关系，有依赖的串行，无依赖的并行 | 多步骤协作任务 |
| 按能力拆 | 不同步骤用不同工具/模型 | 需要多种能力的任务 |

**拆分原则：**
1. 每个子任务的输入输出要清晰明确
2. 子任务之间的依赖关系要标注清楚
3. 每个子任务要能在单次 LLM 调用内完成
4. 要有"重规划"机制——某步结果出乎意料时能调整后续计划

## 8. Agent 记忆机制怎么设计

### 一句话破题

记忆是 Agent 从"一次性问答工具"变成"真正了解你的助手"的分水岭——关键不是存多少，而是**存什么、怎么存、什么时候取**。

### 核心三问

1. 没有记忆的 Agent 有什么问题？
2. 短期记忆和长期记忆有什么本质区别？
3. 什么信息值得存，什么不值得存？

### 类比记忆法

| 记忆类型 | 类比 | 特点 |
|---------|------|------|
| 短期记忆 | 你的工作台 | 随手可见，但空间有限，下班清空 |
| 长期记忆 | 公司档案室 | 容量无限，但要主动去翻才能找到 |
| 实体记忆 | 客户的病历卡 | 结构化记录关键信息，查询快 |

### 深度拆解

**存什么——判断标准：** "这条信息，下次任务开始时如果知道，会让 Agent 做得更好吗？"

| 值得存 | 不值得存 |
|--------|---------|
| 用户偏好和习惯 | 中间推理过程 |
| 任务执行中的关键结论 | 工具返回的原始日志 |
| 外部知识（产品文档、FAQ） | 闲聊内容 |

**怎么存——混合存储是主流：**

| 信息类型 | 存储介质 | 原因 |
|---------|---------|------|
| 非结构化文本 | 向量数据库 | 语义相似度检索 |
| 结构化字段（用户偏好） | 关系数据库/KV | 精确查询，速度快 |
| 整段文档知识 | 向量数据库 + RAG | 语义召回 |

**什么时候取——两种策略结合：**
1. **主动检索**：任务开始前，用任务描述检索相关记忆，注入 system prompt
2. **按需检索**：把"查记忆"封装成 Tool，Agent 执行中自己判断何时检索

### 面试话术

> "Agent 记忆设计的核心是回答三个问题：存什么、怎么存、什么时候取。
>
> 存什么——只存对下次任务有价值的内容：用户偏好、关键结论、外部知识。中间推理过程和原始日志不值得存，存太多反而引入噪音。
>
> 怎么存——混合存储是主流。非结构化文本用向量数据库做语义检索，结构化偏好字段用关系数据库做精确查询。
>
> 什么时候取——主动检索和按需检索结合。Session 开始时主动检索加载用户画像和背景，执行中遇到需要历史数据时按需检索。
>
> 整体形成'读→用→写'三阶段闭环：任务开始读记忆，执行中用记忆，结束后写新记忆。Agent 用得越多，积累越厚。"

## 9. 记忆怎么存、粒度多少、怎么用

### 一句话破题

记忆粒度的核心矛盾是**精度 vs 效率**——存太细检索噪音大，存太粗信息丢失多，需要根据场景找到平衡点。

### 深度拆解

**四层记忆体系（从最短暂到最持久）：**

| 层级 | 载体 | 生命周期 | 访问方式 |
|------|------|---------|---------|
| 感知记忆 | 当次输入 | 单次调用 | 即时访问 |
| 短期记忆 | context window | 一次任务 | 直接读取 |
| 长期记忆 | 向量/关系数据库 | 持久 | 语义检索 |
| 实体记忆 | 结构化存储 | 持久 | 精确查询 |

**长期记忆的三种子类型：**

| 子类型 | 存什么 | 举例 |
|--------|--------|------|
| 情节记忆 | 具体经历 | "上次用户让我写爬虫，遇到反爬，用 Selenium 解决" |
| 语义记忆 | 提炼的通用规律 | "目标网站有 JS 动态渲染时，用浏览器自动化工具" |
| 程序记忆 | 操作流程 SOP | "部署 Flask：虚拟环境→装依赖→gunicorn→nginx" |

**记忆衰减机制：**

```
记忆相关性分数 = 语义相似度 × 时间衰减因子
```

越久远的记忆权重越低，衰减速度可按场景调整：客服场景衰减快（大多一次性对话），法律合规场景衰减慢（历史记录可能很久后被引用）。

**Context Window 管理策略：**

| 策略 | 做法 | 代价 |
|------|------|------|
| 滑动窗口 | 只保留最近 N 轮 | 早期重要信息可能丢失 |
| 摘要压缩 | 用 LLM 把早期对话压缩成摘要 | 丢失细节，额外 LLM 调用 |
| 卸载到长期记忆 | 不常用但重要的信息存入向量库 | 需管理加载/卸载时机 |

## 10. 什么是 Multi-Agent

### 一句话破题

Multi-Agent 就是**多个 Agent 分工协作**完成一个复杂任务，就像公司里不同部门配合做项目——每个 Agent 负责自己擅长的领域，通过通信协议协调。

### 类比记忆法

单 Agent 是一个**全栈工程师**什么都自己干，Multi-Agent 是一个**开发团队**：前端、后端、测试各司其职，通过接口通信协作。

### 深度拆解

**为什么需要多 Agent：**
- 单 Agent 上下文窗口有限，复杂任务信息量太大
- 不同子任务需要不同的工具集和 System Prompt
- 并行执行提高效率
- 单 Agent 角色太多容易"人格分裂"，输出不稳定

**常见架构：**

| 架构 | 说明 | 举例 |
|------|------|------|
| 中心化 | 一个 Master Agent 分配任务给多个 Worker | 项目经理分配任务给组员 |
| 去中心化 | Agent 之间直接通信协作 | 团队成员直接沟通 |
| 层级化 | 多层管理结构 | 总监→经理→工程师 |
| 竞争式 | 多个 Agent 给出方案，选最优 | 头脑风暴 |

---

## 11. 单 Agent 还是多 Agent

### 一句话破题

**能用单 Agent 搞定的就用单 Agent**，多 Agent 是复杂场景的解决方案，不是炫技——通信成本、协调复杂度、调试难度都会指数级上升。

### 深度拆解

| 维度 | 单 Agent | Multi-Agent |
|------|---------|-------------|
| 实现复杂度 | 低 | 高 |
| 通信成本 | 无 | 有，且随 Agent 数量增长 |
| 调试难度 | 相对容易 | 难，错误链路长 |
| 适合任务 | 中等复杂度 | 非常复杂、需要多种专业能力 |
| 并行能力 | 弱 | 强 |

**选型建议：**
1. 先用单 Agent 做，搞不定再考虑多 Agent
2. 任务能拆成相对独立的部分时，考虑多 Agent
3. 不同子任务需要完全不同的工具集和角色设定时，考虑多 Agent

## 12. 记忆压缩有哪些方法

### 一句话破题

记忆压缩就是在"信息太多撑爆上下文"和"压缩太狠丢了关键信息"之间找平衡。

### 深度拆解

| 方法 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| LLM 摘要 | 用大模型把历史对话压缩成摘要 | 保留语义，灵活 | 额外 LLM 调用成本 |
| 滑动窗口 | 只保留最近 N 轮 | 简单高效 | 早期信息直接丢弃 |
| 关键信息提取 | 只保留实体、决策、结论 | 信息密度高 | 可能遗漏上下文 |
| 向量卸载 | 存入向量库，需要时检索 | 不丢信息 | 检索延迟 |
| 分层压缩 | 近期全量→中期摘要→远期只存关键词 | 平衡好 | 实现复杂 |

## 13. 为什么有时候选择手搓 Agent

### 一句话破题

框架是"快速起步"的脚手架，手搓是"精细控制"的车床——**当框架的抽象成为障碍而非助力时，就该手搓了**。

### 深度拆解

**用框架的好处：** 快速起步、社区支持、内置常见模式
**用框架的痛点：**
- 抽象层太多，debug 困难（出错了不知道是哪层的问题）
- 灵活性受限（框架不支持的模式很难实现）
- 版本升级经常 breaking change
- 性能开销（多层封装）

**手搓的好处：**
- 完全控制每一个环节
- 代码简洁直接，没有魔法
- 性能最优，没有框架开销
- debug 容易，所有逻辑都在眼前

**手搓的核心其实就是：**
```python
while not task_done:
    response = llm.generate(messages)  # 1. 调模型
    if response.is_tool_call:          # 2. 判断要不要调工具
        result = execute_tool(response) # 3. 执行工具
        messages.append(result)         # 4. 结果塞回对话
    else:
        task_done = True                # 5. 模型给最终答案了
```

就这五步，框架帮你封装的也是这五步。

## 14. 如何赋予 LLM 规划能力

### 一句话破题

规划能力的本质是让 LLM **在想清楚整体之前不要急着动手**——通过 Prompt 引导、思维链展开、计划-执行分离来实现。

### 深度拆解

| 技术手段 | 做法 | 原理 |
|---------|------|------|
| CoT | Prompt 里加"一步步思考" | 让中间推理显式化，减少累积误差 |
| ToT | 每个节点展开多个分支选最优 | 不一条路走到底，探索多条路径 |
| Plan-and-Execute | 先输出完整计划再执行 | 全局视角，避免局部最优 |
| ReAct + 反思 | 每步做完检查对不对 | 及时纠错，防止错误传播 |

**关键认知：** LLM 本身就有一定的规划能力（特别是大模型），我们需要做的是通过 Prompt 设计和系统架构把这个能力"激发"出来，而不是从零赋予。

## 15. 反思机制怎么实现

### 一句话破题

反思就是让 Agent **做完一步或整个任务后回头审视**——"我做得对不对？结果和预期一致吗？要不要调整？"——本质是给前向推理加一个"回头检查"的能力。

### 类比记忆法

普通 Agent 是**交了卷就走**的学生，有反思机制的 Agent 是**交卷前检查一遍**的学生——检查不保证全对，但能抓住明显的错误。

### 深度拆解

**反思的实现方式：**

```
执行步骤 → 拿到结果 → 反思（LLM 评估质量）→
    如果达标 → 继续下一步
    如果不达标 → 分析问题 → 重新执行或调整计划
```

**反思的层次：**

| 层次 | 反思什么 | 频率 |
|------|---------|------|
| 步骤级反思 | 当前这步结果对不对 | 每步都做（成本高） |
| 阶段级反思 | 这一阶段整体质量如何 | 每个阶段结束做 |
| 任务级反思 | 最终输出是否满足目标 | 任务完成时做 |

**为什么要反思：**
- ReAct 模式没有内置回头检查机制，中间出错一路错
- 反思能发现"看起来完成但质量不行"的问题
- 反思能让 Agent 从错误中学习，调整后续策略

### 面试话术

> "反思机制本质是给前向推理的 Agent 加一个'回头检查'的能力。实现方式是让 LLM 在执行完一步或一个阶段后，评估结果质量——如果达标继续，不达标就分析问题、调整策略。
>
> 反思分三个层次：步骤级（每步检查，成本高但最精细）、阶段级（每阶段结束检查，性价比高）、任务级（最终输出检查，兜底）。
>
> 实际项目里我倾向于阶段级反思——每步都反思 token 消耗太大，只在关键节点做检查，既保证质量又控制成本。"

## 16. 多 Agent 怎么协作和切换

### 一句话破题

多 Agent 协作的核心挑战是**任务分配、通信协议、状态同步**——让对的 Agent 在对的时间做对的事。

### 深度拆解

**协作机制：**

| 机制 | 说明 | 举例 |
|------|------|------|
| 任务路由 | 根据任务类型分给对应 Agent | 用户问代码→给代码Agent，问数据→给数据Agent |
| 结果传递 | 上一个 Agent 的输出是下一个的输入 | 研究Agent输出→写作Agent输入 |
| 动态切换 | 执行中发现当前 Agent 不合适，切换到另一个 | 代码Agent发现需要查文档→切给搜索Agent |
| 竞争选择 | 多个 Agent 给方案，选最优 | 多个Agent各写一段→评审Agent选最好的 |

**A2A 协议（Agent 间通信标准）：**
- Google 2025年4月提出，6月捐给 Linux 基金会
- 每个 Agent 有一张"名片"（Agent Card）：能做什么、正在做什么、需要什么输入
- 其他 Agent 读了名片就知道怎么协作
- MCP 管 Agent↔工具，A2A 管 Agent↔Agent

**动态切换的关键：** 要有"兜底"机制——切换失败时不能让整个系统卡死。

## Agent 篇总结

### 知识地图

```
Agent 是什么？
├── 本质：自主闭环（感知→规划→行动→再感知）
├── 四大组件：LLM + 工具 + 记忆 + 规划
└── 和 LLM 的区别：自主性 + 能行动 + 有闭环

怎么设计？
├── 范式：ReAct（灵活）vs Plan-Execute（全局）vs Reflection（纠错）
├── 实践：混合模式，大模型规划+小模型执行
└── 记忆：短期（context）+ 长期（向量库），读→用→写闭环

怎么扩展？
├── 多 Agent：分工协作，通信协议（A2A）
├── 记忆压缩：摘要/滑动窗口/向量卸载
└── 手搓 vs 框架：控制力 vs 开发效率
```

### 高频面试题 TOP 5

1. **"什么是 Agent？和普通大模型有什么区别？"** → 自主闭环，三件事（自主规划、能行动、有反馈）
2. **"Agent 的核心组件有哪些？"** → 四大件：LLM、工具、记忆、规划
3. **"ReAct 是什么？怎么实现的？"** → 思考→行动→观察循环，代码驱动
4. **"Agent 记忆怎么设计？"** → 存什么/怎么存/什么时候取 + 读→用→写闭环
5. **"ReAct 和 Plan-and-Execute 怎么选？"** → 混合用，规划用大模型执行用小模型
