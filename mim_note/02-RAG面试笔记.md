## 1. RAG 是什么、完整流程怎么跑

### 一句话破题

RAG = **Retrieval-Augmented Generation**（检索增强生成），在 LLM 生成答案之前，先去外部知识库检索相关内容，把检索结果和用户问题一起交给 LLM——本质就是**开卷考试**。

### 核心三问

1. RAG 和传统搜索引擎有什么区别？（搜索引擎只检索不生成）
2. RAG 和微调有什么区别？（微调改参数，RAG 不改）
3. 完整的 RAG 流程分几个阶段？（离线建库 + 在线检索生成）

### 类比记忆法

| 方式 | 类比 | 特点 |
|------|------|------|
| 纯 LLM | 闭卷考试 | 只靠脑子里的知识，考完就忘不了也学不了新的 |
| 微调 | 重新上个大学 | 把新知识学进脑子里，成本高但内化了 |
| RAG | 开卷考试 | 不改脑子，给一本参考书，答题时翻书 |

### 深度拆解

**完整流程分两个阶段：**

```
【离线阶段】只做一次，建好知识库反复用
文档加载 → 文档切割 → Embedding向量化 → 入向量数据库

【在线阶段】每次用户提问都跑一遍
用户提问 → Query改写 → 向量检索(粗排) → Rerank(精排) → 拼接Prompt → LLM生成
```

**离线阶段四步：**

| 步骤 | 干什么 | 关键点 |
|------|--------|--------|
| 文档加载 | 把 PDF/Word/网页等读进来 | 支持几十种格式，用 LlamaIndex/LangChain 的 Loader |
| 文档切割 | 把长文档切成小块(chunk) | 大小 500~1000 token，重叠 100 token 防语义断裂 |
| Embedding | 把文本块转成高维向量 | 语义相近的文本在向量空间里距离近 |
| 入库 | 向量+原文存入向量数据库 | 支持千万级向量的快速相似度搜索 |

**在线阶段四步：**

| 步骤 | 干什么 | 关键点 |
|------|--------|--------|
| Query改写 | 把口语化提问改成适合检索的形式 | 补充上下文，消除指代 |
| 向量检索(粗排) | 用 Query 向量找最相似的 Top-K chunk | 快（毫秒级），但会混入"看着近但不相关"的内容 |
| Rerank(精排) | 用 Cross-Encoder 深度评估相关性，重排序 | 更准但更慢，只对 Top-20 做精排，留 Top-3~5 |
| 生成 | 检索结果+用户问题拼成 Prompt 交给 LLM | 明确告诉 LLM"只根据资料回答，没有就说不知道" |

**粗排 vs 精排的类比：**
- 粗排 = 在书架上快速扫一眼，把看着可能相关的书都抽出来
- 精排 = 一本本翻开读目录，确认哪些真正有用

### 面试话术

> "RAG 全称是检索增强生成，核心解决的是 LLM 知识冻结的问题——模型训练完知识就固定了，遇到私有数据或最新信息答不上来。RAG 的做法是在生成答案前先去外部知识库检索相关内容，把检索结果和问题一起交给 LLM，本质就是开卷考试。
>
> 完整流程分两个阶段。离线阶段只做一次：文档加载→切割→向量化→入库。在线阶段每次提问都跑：Query改写→向量检索粗排→Rerank精排→拼接Prompt→LLM生成。
>
> RAG 的核心价值有两个：知识可热更新（加文档就行不用重新训练），答案可溯源（每条回答能追溯到来源）。这也是企业落地 AI 问答首选 RAG 的原因。"

### 雷区雷达

| 坑 | 正确理解 |
|----|---------|
| 把 RAG 等同于搜索引擎 | 搜索引擎只检索不生成，RAG 有 LLM 生成环节 |
| 把 RAG 和微调搞混 | 微调改模型参数，RAG 不动模型 |
| 忽略文档切割和 Rerank | 这些工程细节恰恰是面试官追问的重点 |

## 2. RAG 主要解决什么问题

### 一句话破题

RAG 解决三个问题：**知识冻结**（训练完就不更新了）、**私有数据盲区**（公司内部文档 LLM 不知道）、**幻觉**（LLM 爱编）——靠外部检索来补。

### 深度拆解

| 问题 | 具体表现 | RAG 怎么解决 |
|------|---------|-------------|
| 知识冻结 | 训练数据截止日期后的事不知道 | 接外部知识库，实时检索最新信息 |
| 私有数据盲区 | 公司内部文档 LLM 完全不知道 | 把私有文档建进知识库 |
| 幻觉 | LLM 会编造看似合理但错误的信息 | 让 LLM 基于检索到的资料回答，减少编造 |

### 面试话术

> "RAG 主要解决三个问题。第一，知识冻结——LLM 训练完知识就固定了，RAG 通过外部检索获取最新信息。第二，私有数据盲区——公司内部文档 LLM 不知道，RAG 把这些文档建进知识库。第三，幻觉——通过让 LLM 基于检索到的资料回答，并且明确告诉它'资料里没有就说不知道'，有效抑制编造。"

## 3. RAG 和微调怎么选

### 一句话破题

微调是把知识**学进脑子里**（改参数），RAG 是给模型**一本参考书**（不改参数）——**不是二选一，而是互补**。

### 类比记忆法

| 维度 | 微调 | RAG |
|------|------|-----|
| 类比 | 为了记住一条新闻重新上大学 | 查字典/开卷考试 |
| 知识存储 | 写进模型参数 | 外部知识库 |
| 更新成本 | 高，需重新训练 | 低，加文档即可 |
| 更新方式 | 重新训练 | 热更新 |
| 知识溯源 | 不可溯源 | 可追溯到来源 |
| 适合 | 改变模型风格/能力 | 补充事实知识 |

### 深度拆解

**选型决策树：**

```
你需要的是事实知识吗？
├── 是 → RAG（更新快、可溯源）
│   └── 知识量特别大且不常变？→ 考虑微调
└── 否 → 你需要改变模型的风格/能力吗？
    ├── 是 → 微调
    └── 否 → 可能只需要 Prompt Engineering
```

**实际项目中的组合：** 先 RAG 解决知识问题，如果模型输出风格/格式不满意，再叠加微调。

### 面试话术

> "微调和 RAG 不是对立的，是互补的。微调改模型参数，适合改变模型风格和能力；RAG 不改参数，适合补充事实知识。
>
| 维度 | 微调 | RAG |
|------|------|-----|
| 更新成本 | 高 | 低 |
| 知识溯源 | 不可 | 可溯源 |
| 适合 | 风格/能力 | 事实知识 |

> 实际项目中我倾向于先上 RAG——成本低、可溯源、热更新。如果 RAG 解决不了的问题（比如模型输出风格不满足要求），再考虑微调。很多生产系统是 RAG + 轻量微调组合使用。"

## 4. 文档怎么切、粒度多大

### 一句话破题

文档切割是 RAG 离线阶段最影响效果的环节——切太大信息太杂检索不精确，切太小语义不完整检索不到。

### 核心三问

1. 为什么不能直接存整篇文档？
2. chunk 大小和 overlap 怎么定？
3. 有哪些切割策略？

### 类比记忆法

把文档想象成一本**菜谱**：
- 不切 → 查"红烧肉怎么做"时把整本书翻一遍，信息太多找不到
- 切成单字 → "红""烧""肉"完全失去语义
- 按"一道菜"为单位切 → 完美，每块是一个完整的知识单元

### 深度拆解

**为什么不能存整篇：**
1. Embedding 模型有输入长度限制（几百到几千 token）
2. 整篇文档压缩成一个向量，细节信息被"平均掉"——你问"这道菜怎么样"，回答"中国菜整体偏咸"，具体哪道菜咸全丢了

**推荐参数：**

| 参数 | 推荐值 | 原因 |
|------|--------|------|
| chunk 大小 | 500~1000 token | 太大信息杂，太小语义不完整 |
| overlap | 前后各重叠 ~100 token | 避免把完整语义从中间切断 |

**切割策略：**

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| 固定长度 | 按 token 数固定切 | 简单但可能切断语义 |
| 按分隔符 | 按段落/标题/句子切 | 保持语义完整性 |
| 递归切割 | 先按大分隔符切，太长再按小的切 | 兼顾语义和长度 |
| 语义切割 | 用模型判断语义边界 | 效果最好但成本高 |
| 按文档结构 | 按 Markdown 标题层级切 | 结构化文档 |

### 面试话术

> "文档切割是 RAG 里最影响效果的环节。不能存整篇文档，一是因为 Embedding 模型有输入长度限制，二是整篇压缩成一个向量细节信息会被平均掉。
>
> 通常按 500~1000 token 切，前后重叠 100 token 防止语义断裂。切割策略上，最常用的是递归切割——先按大分隔符（如标题）切，太长再按小分隔符（如段落、句子）切，兼顾语义完整性和长度控制。
>
> 对于结构化文档（如 Markdown），按标题层级切效果更好，因为天然保持了语义边界。"

## 5. 怎么避免语义被切断

### 一句话破题

语义切断就是一刀切在了一句话或一段逻辑的中间——解决方案是**重叠切割 + 智能分割 + 元数据保留**。

### 深度拆解

| 方法 | 做法 | 效果 |
|------|------|------|
| 重叠切割 | 相邻 chunk 重叠 100~200 token | 切断的内容在相邻 chunk 里有完整版本 |
| 按语义边界切 | 在段落/句子结束处切 | 天然保持语义完整 |
| 递归切割 | 优先按大结构切，不行再切小 | 尽量保持最大粒度的语义单元 |
| 元数据保留 | 每个 chunk 标注来源文档、章节、页码 | 检索时可以按元数据过滤 |
| 父子文档 | 切成小 chunk 检索，找到后返回所属的大段落 | 检索精确 + 上下文完整 |

## 6. Embedding 模型怎么选

### 一句话破题

Embedding 就是把"意思"编码成数学坐标——意思越近坐标越近，不是在匹配关键词，而是在**比较意思相不相近**。

### 类比记忆法

Embedding 像一个"语义 GPS"——每段文字在坐标系里有一个位置，意思相近的文字位置就挨着。检索就是算"谁离我最近"。

**举例：** "苹果手机怎么截图"和"iPhone 如何截屏"——用词完全不一样，但意思一样，所以向量距离非常近。

### 深度拆解

**Embedding 模型怎么做到的：**
1. 训练时看了海量文本
2. 学会识别哪些句子表达同一个意思
3. 用对比学习调整权重
4. 让"意思相近的文本"在输出空间里靠近

**选型考虑因素：**

| 因素 | 说明 |
|------|------|
| 语言支持 | 中文/英文/多语言 |
| 维度 | 常见 768/1024/1536 维，维度高表达能力强但存储和计算成本高 |
| 性能 | MTEB 排行榜参考，但要看自己业务场景的效果 |
| 部署方式 | API 调用 vs 本地部署 |
| 成本 | API 按 token 计费，本地部署需要 GPU |

**常见模型：**
- OpenAI text-embedding-3-small/large
- BGE 系列（中文效果好，开源）
- E5 系列
- Cohere embed

## 7. Embedding 有哪些算法

### 一句话破题

Embedding 算法的核心思想都是**对比学习**——让相似的文本靠近，不相似的远离，区别在于"怎么定义相似"和"网络结构"。

### 深度拆解

| 算法类别 | 核心思路 | 代表模型 |
|---------|---------|---------|
| 对比学习 | 正样本对拉近，负样本对推远 | SimCSE, BGE |
| 编码器+池化 | BERT 类编码后取平均/CLS 作为向量 | Sentence-BERT |
| 自监督 | 利用数据自身的结构做监督 | E5 |
| 指令微调 | 让模型理解"用这个角度来编码" | BGE, GTE |

## 8. 向量数据库怎么选

### 一句话破题

向量数据库是专门优化了高维向量存储和相似度搜索的数据库——选型核心看**数据量级、性能要求、部署方式**。

### 深度拆解

| 数据库 | 特点 | 适合场景 |
|--------|------|---------|
| Chroma | 轻量，Python 原生，易上手 | 原型开发、小数据量 |
| Milvus | 分布式，高性能，支持十亿级向量 | 生产环境、大规模 |
| Qdrant | Rust 写的，性能好，过滤能力强 | 需要复杂元数据过滤 |
| Weaviate | 内置多种检索方式 | 需要混合检索 |
| FAISS | Facebook 的库，不是数据库 | 纯向量检索，自己管理存储 |
| pgvector | PostgreSQL 扩展 | 已有 PG 基础设施，不想引入新组件 |

**选型维度：**

| 维度 | 要问自己 |
|------|---------|
| 数据量 | 百万级？十亿级？ |
| 性能 | 延迟要求多少？QPS 多少？ |
| 部署 | 云服务还是自建？ |
| 过滤 | 需要按元数据过滤吗？ |
| 生态 | 和现有技术栈兼容吗？ |

## 9. 向量数据库实践经验

### 一句话破题

生产环境的向量数据库，**索引选择、内存管理、批量写入**是三大性能瓶颈。

### 深度拆解

**索引算法选择：**

| 索引 | 原理 | 特点 |
|------|------|------|
| Flat | 暴力搜索 | 100%准确，但慢，适合小数据 |
| IVF | 先聚类再在类内搜索 | 牺牲少量精度换大幅加速 |
| HNSW | 图结构索引 | 查询快，构建慢，内存大 |
| PQ | 乘积量化压缩 | 内存占用小，精度有损 |

**性能优化经验：**
- 百万级以下用 Flat 就行，别过度优化
- HNSW 是生产环境最常用的，速度和精度平衡好
- 批量写入用异步，别一条一条插
- 预加载索引到内存，避免冷查询
- 监控召回率，索引参数不对会导致召回率下降

## 10. 在线阶段怎么跑

### 一句话破题

在线阶段就是"用户提问→改写问题→检索→精排→生成"的流水线，每一步都有优化空间。

### 深度拆解

```
用户提问 "上次说的方案怎么样"
    ↓ Query改写（补充上下文："用户上次讨论的XX方案进展如何"）
    ↓ 向量检索（找最相似的 Top-20 chunk）
    ↓ Rerank精排（Cross-Encoder 重新排序，留 Top-5）
    ↓ 拼接Prompt（"根据以下资料回答用户问题：[资料] 问题：..."）
    ↓ LLM生成答案
```

每个环节都可能出问题：
- Query 没改写好 → 检索方向就错了
- 粗排召回率低 → 精排再好也没用
- Prompt 没约束好 → LLM 可能不基于资料回答

## 11. 向量检索和关键词检索的区别

### 一句话破题

向量检索比"意思"，关键词检索比"字面"——**向量检索能理解同义词，关键词检索能精确匹配专有名词**，生产环境两者结合用。

### 深度拆解

| 维度 | 向量检索 | 关键词检索(BM25) |
|------|---------|-----------------|
| 原理 | 语义相似度 | 词频/逆文档频率 |
| 优势 | 理解同义词、近义词 | 精确匹配专有名词、代码、ID |
| 劣势 | 可能召回"意思近但不相关"的 | 不理解语义，同义词匹配不到 |
| 举例 | "怎么截图"能匹配"如何截屏" | "iPhone 15"能精确匹配 |

**为什么需要结合：**
- 用户搜"错误码 ERR_4021" → 向量检索可能匹配到其他错误码，关键词检索能精确命中
- 用户搜"怎么让电脑更快" → 关键词检索匹配不到"优化系统性能"，向量检索可以

## 12. 怎么润色用户的 Query

### 一句话破题

用户的提问往往是口语化、模糊的、带上下文指代的——Query 改写就是让 LLM 把"人话"翻译成"检索系统能理解的话"。

### 深度拆解

**为什么需要改写：**

| 问题 | 举例 |
|------|------|
| 口语化 | "那玩意儿怎么用" → 指代不明 |
| 带上下文指代 | "上次说的那个方案" → 离开对话上下文没法检索 |
| 太短 | "退货" → 缺少关键信息 |
| 太长 | 一段话 → 需要提取核心问题 |

**改写策略：**

| 策略 | 做法 | 举例 |
|------|------|------|
| 指代消解 | 从对话历史补全指代 | "它怎么样" → "iPhone 15 性能怎么样" |
| 查询扩展 | 加上同义词/相关词 | "退货" → "退货流程 退款政策" |
| 子问题拆分 | 复杂问题拆成多个子问题 | "对比A和B的价格和性能" → 4个子查询 |
| HyDE | 让 LLM 先生成一个假设性答案，用答案去检索 | 先生成"iPhone 15 性能很好..."再用这段检索 |

## 13. 什么是多路召回

### 一句话破题

多路召回就是**用多种检索方式同时找，再合并结果**——向量检索+关键词检索+知识图谱检索，各有所长，互补使用。

### 深度拆解

```
用户Query
    ├── 向量检索 → Top-K 结果
    ├── 关键词检索(BM25) → Top-K 结果
    ├── 知识图谱检索 → Top-K 结果
    └── 其他检索方式 → Top-K 结果
         ↓
    合并去重 → Rerank精排 → 最终Top-N
```

**为什么需要多路：**
- 向量检索擅长语义匹配，但可能召回"意思近但不相关"的
- 关键词检索精确匹配专有名词，但不理解同义词
- 知识图谱擅长关系推理，但覆盖面有限
- 多路合并后用 Rerank 统一排序，取各家之长

**合并策略：**
- 简单合并：直接拼接去重
- 加权融合：不同路给不同权重
- Rerank 统一排序：最常用，效果最好

---

## 14. 检索优化有哪些策略

### 一句话破题

检索优化贯穿 RAG 全流程——从 Query 改写、多路召回、Rerank 精排，到元数据过滤、上下文压缩，每个环节都有招。

### 深度拆解

| 优化环节 | 策略 | 效果 |
|---------|------|------|
| Query 端 | 查询改写、扩展、子问题拆分 | 让检索方向更准确 |
| 检索端 | 多路召回（向量+关键词+图谱） | 提高召回率 |
| 排序端 | Rerank 精排 | 过滤不相关结果，提高精确率 |
| 过滤端 | 元数据过滤（时间、来源、类别） | 缩小检索范围，提高精度 |
| 上下文端 | 压缩/去重/重排检索结果 | 减少噪音，节省 token |
| Chunk 端 | 优化切割策略、父子文档 | 提高检索单元质量 |

## 15. 有哪些高级 RAG 范式

### 一句话破题

基础 RAG 是"检索一次→生成一次"，高级 RAG 在此基础上加**迭代检索、自适应检索、多跳推理**等能力。

### 深度拆解

| 范式 | 核心思路 | 适合场景 |
|------|---------|---------|
| Naive RAG | 检索一次→生成一次 | 简单问答 |
| Advanced RAG | 检索前后加优化（改写、Rerank） | 通用场景 |
| Modular RAG | 模块化，可灵活组合检索-生成模块 | 复杂系统 |
| Self-RAG | 模型自己判断要不要检索、检索结果好不好 | 需要灵活判断 |
| CRAG | 检索质量不好时自动切换到网页搜索 | 需要兜底 |
| RAG-Fusion | 多角度生成 Query 并行检索再融合 | 复杂问题 |
| FLARE | 边生成边检索，遇到不确定的地方才检索 | 长文生成 |

## 16. 什么时候用图数据库

### 一句话破题

当检索需要**关系推理**（"A公司的CTO所在公司做什么业务"）时，向量数据库做不到——这种多跳关系查询正是图数据库的强项。

### 深度拆解

**向量数据库的局限：** 一条一条存取，每条记忆之间独立无关联。无法回答"A公司的CTO所在公司做什么业务"这种需要跨实体关系推理的问题。

**图数据库的解法：** 用"实体→关系→实体"的三元组存储。

```
用户A → 担任CTO → 公司B
公司B → 主营业务 → 云计算
公司B → 成立于 → 2015年
```

查"A所在公司做什么业务"：从A出发沿"担任CTO"找到B，再沿"主营业务"找到云计算——两跳拿到答案。

**GraphRAG（图增强 RAG）：**
- 对话中用 LLM 自动提取实体和关系存入图谱
- 检索时先用向量检索拿候选，再用图谱补充关联信息
- 适合需要关系推理的场景（知识图谱问答、多跳推理）

## 17. 怎么规避 RAG 中的幻觉

### 一句话破题

RAG 中的幻觉主要来自两个地方：**检索不到相关内容时 LLM 硬编**，和**检索到了但 LLM 不基于资料回答**——解法是"检索兜底 + Prompt 约束 + 输出校验"。

### 深度拆解

| 策略 | 做法 | 原理 |
|------|------|------|
| Prompt 约束 | "只根据提供的资料回答，资料里没有就说不知道" | 直接限制 LLM 编造 |
| 检索质量保证 | Rerank 精排 + 相似度阈值过滤 | 确保给 LLM 的资料是相关的 |
| 兜底机制 | 检索结果为空或相似度太低时，直接返回"没有找到相关信息" | 不让 LLM 在没有资料时硬编 |
| 引用标注 | 要求 LLM 标注每句话的来源 | 便于人工校验，也迫使 LLM 基于资料回答 |
| 输出校验 | 用另一个 LLM 检查输出是否与检索资料一致 | 最后一道防线 |

## 18. 怎么量化 RAG 效果

### 一句话破题

RAG 效果评估要分两段：**检索质量**（找得准不准）和**生成质量**（答得好不好），不能只看最终答案。

### 深度拆解

**检索质量指标：**

| 指标 | 衡量什么 |
|------|---------|
| Recall@K | 前 K 个结果中包含正确答案的比例 |
| Precision@K | 前 K 个结果中相关的比例 |
| MRR | 正确答案的排名倒数 |
| NDCG | 考虑排名位置的加权相关性 |

**生成质量指标：**

| 指标 | 衡量什么 |
|------|---------|
| Faithfulness | 答案是否忠于检索到的资料（不编造） |
| Answer Relevancy | 答案是否回答了用户问题 |
| Context Precision | 检索到的上下文是否精确相关 |
| Context Recall | 是否检索到了所有需要的上下文 |

**评估方法：**
- 人工标注：最准但成本高
- LLM-as-Judge：用大模型评估大模型，成本低但有不确定性
- RAGAS 框架：自动化评估 RAG 的常用工具
- A/B 测试：线上对比不同方案

## 19. 知识库怎么动态更新

### 一句话破题

知识库不是建好就不管了——需要**增量更新、版本管理、失效检测**，保证检索到的信息是最新的。

### 深度拆解

| 更新场景 | 做法 |
|---------|------|
| 新增文档 | 切割→向量化→入库 |
| 修改文档 | 删除旧 chunk → 重新切割入库 |
| 删除文档 | 按 metadata 批量删除对应向量 |
| 定期全量重建 | 定期从头重建，避免增量更新累积的碎片 |

**关键挑战：**
- 增量更新 vs 全量重建的权衡
- 文档修改后旧向量要及时失效
- 更新过程中不能影响在线检索
- 版本管理（回滚能力）

## 20. RAG 落地最难的是什么

### 一句话破题

RAG 落地最难的**不是技术实现，而是效果调优**——检索质量、文档质量、评估体系，每一样都是脏活累活。

### 深度拆解

| 难点 | 为什么难 | 怎么应对 |
|------|---------|---------|
| 文档质量参差不齐 | 真实文档格式混乱、内容重复、信息过时 | 建文档预处理流水线，清洗+去重+质量评估 |
| 切割策略难调 | 不同文档类型需要不同策略 | 按文档类型分类，每类用不同切割策略 |
| 检索效果不稳定 | 有些问题检索准，有些完全检索不到 | 多路召回 + Rerank + 持续优化 |
| 评估体系缺失 | 没有标注数据，不知道效果好不好 | 先建评估集，定期跑评估，持续迭代 |
| 幻觉无法完全消除 | LLM 始终有可能编造 | Prompt 约束 + 输出校验 + 兜底机制 |
| 用户问法千变万化 | 同一个问题有无数种问法 | Query 改写 + 扩展 + 同义词库 |

### 面试话术

> "我认为 RAG 落地最难的不是技术实现——框架和工具已经很成熟了——而是效果调优。
>
> 具体来说有几个难点：第一，文档质量参差不齐，真实文档格式混乱、内容重复、信息过时，需要建预处理流水线。第二，切割策略难调，不同文档类型需要不同策略，没有银弹。第三，评估体系缺失，没有标注数据就不知道效果好不好，所以我倾向于先建评估集再迭代。第四，用户问法千变万化，需要 Query 改写和扩展来覆盖。
>
> 这些都不是靠一个算法能解决的，需要持续的工程迭代。"

## RAG 篇总结

### 知识地图

```
RAG 是什么？
├── 本质：检索增强生成（开卷考试）
├── 解决：知识冻结、私有数据盲区、幻觉
└── vs 微调：不改参数、可热更新、可溯源

怎么建库？（离线）
├── 文档加载 → 切割(500-1000 token) → Embedding → 入库
├── 切割策略：递归切割最常用，按结构切更好
└── Embedding：把"意思"编码成坐标

怎么检索？（在线）
├── Query改写 → 向量检索(粗排) → Rerank(精排) → 生成
├── 多路召回：向量+关键词+图谱
└── 优化：改写、扩展、过滤、压缩

怎么落地？
├── 防幻觉：Prompt约束 + 检索兜底 + 输出校验
├── 评估：检索质量 + 生成质量，分开评
└── 最难：效果调优（文档质量、切割、评估体系）
```

### 高频面试题 TOP 5

1. **"什么是 RAG？完整流程是什么？"** → 开卷考试，离线建库+在线检索生成
2. **"文档怎么切？为什么不能存整篇？"** → 500-1000 token，存整篇信息会被平均掉
3. **"RAG 和微调怎么选？"** → RAG 补知识，微调改能力，互补使用
4. **"怎么优化检索效果？"** → Query改写+多路召回+Rerank+元数据过滤
5. **"RAG 落地最难的是什么？"** → 不是技术实现，是效果调优
