## Q1：什么是Function Calling？原理是什么？

### 一句话破题

Function Calling是大模型「读懂工具说明书、生成结构化调用指令」的能力——不是模型自己去执行函数，而是模型输出一段JSON，告诉调用方「该用哪个函数、传什么参数」。

### 核心三问

**Q：模型会直接执行函数吗？**
不会。模型只负责「决策」——选哪个函数、填什么参数，输出的是JSON文本。真正执行函数的是应用代码。

**Q：模型怎么知道有哪些函数可用？**
开发者在API请求中传入函数描述（名字、参数说明、用途），模型根据这些描述判断是否需要调用。

**Q：Function Call的输出格式是什么？**
结构化JSON，包含函数名和参数键值对，例如：`{"name": "get_weather", "arguments": {"city": "上海"}}`。

### 类比记忆

> 🎯 **类比：点餐系统**
>
> 你（大模型）坐在包间里看菜单（函数描述），你看完菜单决定点哪个菜，然后写一张点餐单（JSON）递给服务员（应用代码）。服务员拿着单子去厨房（执行函数），厨房做好菜端回来，服务员再把菜（函数返回值）给你看，你根据菜的味道继续决定下一步。
>
> **关键点：你从不进厨房，你只写单子。**

### 深度拆解

**完整调用流程（四步循环）：**

```
① 用户提问 → "上海明天天气怎么样？"
② 模型决策 → 检查可用函数列表，发现get_weather匹配
   输出: {"name":"get_weather","arguments":{"city":"上海","date":"tomorrow"}}
③ 应用执行 → 调用真实天气API，拿到结果: {"temp":28,"weather":"晴"}
④ 模型总结 → 把函数结果融入回答: "上海明天晴，气温28°C"
```

**函数描述的写法（OpenAI格式）：**

```json
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "获取指定城市的天气预报",
    "parameters": {
      "type": "object",
      "properties": {
        "city": {"type": "string", "description": "城市名"},
        "date": {"type": "string", "description": "日期，如today/tomorrow"}
      },
      "required": ["city"]
    }
  }
}
```

**模型是怎么「学会」做这件事的？**

不是靠prompt临时教，而是训练阶段就注入了这种能力：
1. **SFT阶段**：用大量「用户问题 + 正确函数调用」的标注数据微调，让模型学会识别「该不该调函数」和「调哪个函数」
2. **RLHF阶段**：用奖励模型优化函数选择的准确性和参数填写的正确性
3. 不同模型的Function Calling能力差异很大——GPT-4、Claude等闭源模型经过专门训练，能力很强；小模型往往需要额外微调

### 面试话术

> "Function Calling本质上是模型的一种结构化输出能力。模型本身不执行任何函数，它做的事情是：阅读开发者提供的函数描述，判断当前用户问题是否需要调用函数，如果需要就输出一段包含函数名和参数的JSON。应用代码拿到这段JSON后执行真实函数，把结果再喂回模型做最终回答。
>
> 这个能力不是靠prompt临时实现的，而是在模型训练阶段通过SFT和RLHF专门注入的。函数描述的质量直接影响调用准确率——description要写清楚用途，参数说明要写清楚格式和约束。"

### 雷区雷达

| 雷区 | 说明 |
|:---:|------|
| ❌ "模型直接执行了函数" | 模型只输出JSON，不执行任何代码 |
| ❌ "Function Calling就是写prompt" | 核心是训练阶段注入的能力，不只是prompt工程 |
| ❌ "参数随便描述就行" | 参数description的质量直接影响调用成功率 |
| ⚠️ 多函数选择 | 模型可能选错函数或填错参数，生产环境要做参数校验和错误兜底 |

## Q2：LLM是如何学会调用外部工具的？

### 一句话破题

LLM调工具的能力来自三个层次的叠加：预训练阶段积累的「世界知识」、SFT阶段学会的「调工具范式」、RLHF阶段优化的「调对工具」。

### 核心三问

**Q：预训练阶段和工具调用有关系吗？**
有。预训练让模型理解了「什么是天气」「什么是搜索」这些概念，没有这些基础知识，模型连函数描述都看不懂。

**Q：SFT阶段具体做了什么？**
构造大量「用户问题 → 正确的函数调用JSON」样本，让模型学会：看到什么问题该调什么函数、参数怎么从用户话里提取。

**Q：为什么不只用Prompt？**
纯prompt方式（如ReAct）依赖模型的指令遵循能力，小模型容易出错。训练方式把工具调用变成了模型的「肌肉记忆」，更稳定可靠。

### 类比记忆

> 🎯 **类比：培养一个翻译官**
>
> - **预训练** = 上大学学外语（积累语言基础和世界知识）
> - **SFT** = 实习时学翻译流程规范（学会按格式输出）
> - **RLHF** = 老师纠正翻译错误，告诉什么翻得好什么翻得差（优化质量）
>
> 三步缺一不可：没上过大学看不懂原文，没实习过不知道翻译格式，没纠正过容易翻错。

### 深度拆解

**三层能力构建：**

| 层次 | 阶段 | 学到什么 | 类比 |
|:---:|:---:|------|------|
| 第一层 | 预训练 | 理解概念、世界知识 | 学外语 |
| 第二层 | SFT | 函数调用格式、参数提取 | 学翻译流程 |
| 第三层 | RLHF | 选对函数、填对参数、拒绝不该调的 | 质量优化 |

**SFT数据构造示例：**

```
输入: "帮我查一下北京到上海的高铁票"
输出: {"name":"search_train","arguments":{"from":"北京","to":"上海","type":"高铁"}}

输入: "今天天气真好啊"
输出: null  （不需要调用函数，直接闲聊回复）
```

关键训练样本类型：
1. **正例**：需要调函数的问题 + 正确的JSON
2. **负例**：不需要调函数的闲聊 → 输出null
3. **多选**：多个函数可选时选对的那个
4. **参数提取**：从自然语言中准确提取参数值
5. **多轮调用**：前一个函数的结果影响后一个函数的选择

### 面试话术

> "LLM调工具的能力是分三层构建的。预训练阶段让模型具备了理解函数描述的基础知识。SFT阶段用大量标注数据教会模型函数调用的格式和参数提取方式，这是最关键的一步。RLHF阶段进一步优化，让模型在多个候选函数中选得更准、参数填得更对。
>
> 纯prompt方式如ReAct也能实现工具调用，但依赖模型的指令遵循能力，小模型容易出格式错误或选错函数。训练方式把工具调用内化为模型能力，稳定性好得多。"

### 雷区雷达

| 雷区 | 说明 |
|:---:|------|
| ❌ "工具调用完全靠prompt" | 核心靠训练注入，prompt只是传递函数描述 |
| ⚠️ 忽略负例训练 | 只训练「该调函数」的样本，模型会对闲聊也尝试调函数 |
| ⚠️ 小模型FC能力差 | 开源小模型未经专门训练，FC能力很弱，需要额外微调 |

## Q3：Function Call能力是怎么训练出来的？

### 一句话破题

Function Call训练 = SFT教会格式 + RLHF优化选择，核心难点在于构造高质量的「问题-函数调用」配对数据。

### 类比记忆

> 🎯 **类比：驾校培训**
>
> - SFT = 教练手把手教你怎么打方向盘、踩油门（学会基本操作）
> - RLHF = 教练在你开歪了的时候纠正你，开好了表扬你（优化驾驶质量）
> - 训练数据 = 各种路况场景（直道、弯道、坡道、雨天...）
>
> 场景覆盖越全，你上路越不怕。

### 深度拆解

**训练Pipeline：**

```
Step 1: 构造训练数据
  ├─ 自动生成：用GPT-4生成「问题→函数调用」样本
  ├─ 人工标注：专家写高质量样本
  └─ 数据增强：同义改写、参数变体、多函数场景

Step 2: SFT（监督微调）
  ├─ 输入：用户问题 + 函数描述列表
  ├─ 目标输出：正确的函数调用JSON 或 null
  └─ 损失函数：标准交叉熵

Step 3: RLHF（强化学习优化）
  ├─ 奖励模型：评估函数选择的合理性和参数的正确性
  ├─ 正奖励：选对了函数、参数填对了
  ├─ 负奖励：选错函数、参数缺失、不该调却调了
  └─ 优化目标：最大化期望奖励
```

**关键训练数据类型：**

| 数据类型 | 占比 | 作用 |
|:---:|:---:|------|
| 单函数正例 | 40% | 学会基本的函数选择和参数提取 |
| 闲聊负例 | 20% | 学会「不该调函数时不调」 |
| 多函数选择 | 20% | 学会在多个候选中选对 |
| 多轮调用 | 10% | 学会根据上一步结果决定下一步 |
| 边界case | 10% | 参数缺失、歧义、无关函数干扰 |

### 面试话术

> "Function Call的训练分两步。SFT阶段用大量「问题-函数调用」配对数据做监督微调，让模型学会输出格式和参数提取，关键是要覆盖正例、负例、多选、多轮等场景。RLHF阶段用奖励模型优化，奖励选对函数和填对参数的行为，惩罚错误调用。训练数据的质量和场景覆盖度是决定模型FC能力的核心因素。"

## Q4：什么是MCP（模型上下文协议）？核心内容是什么？

### 一句话破题

MCP（Model Context Protocol）是Anthropic提出的开放协议，本质上是一套「AI应用连接外部工具/数据源的USB接口标准」——让AI应用不需要为每个工具单独写适配代码，接上MCP Server就能用。

### 核心三问

**Q：MCP解决什么问题？**
「M×N问题」——M个AI应用 × N个工具，需要写M×N套对接代码。MCP把它变成M+N：应用实现一次MCP客户端，工具实现一次MCP服务端，任意组合。

**Q：MCP是协议还是框架？**
是协议（Protocol），定义了通信格式和交互规则，不是具体代码框架。任何人可以按协议实现自己的MCP Server。

**Q：MCP和Function Calling冲突吗？**
不冲突。MCP是「工具发现和接入协议」，Function Calling是「模型决策调哪个工具」。MCP负责把工具能力告诉模型，FC负责模型决定怎么调。

### 类比记忆

> 🎯 **类比：USB标准**
>
> 在USB出现前，键盘用PS/2口、打印机用并口、鼠标用串口，每个设备一个专用接口。USB统一了接口标准，任何设备只要做成USB接口就能插上电脑用。
>
> MCP就是AI领域的USB——在MCP之前，每个AI应用要对接每个工具都得写专门的适配代码；有了MCP，工具做成MCP Server，应用实现MCP Client，即插即用。

### 深度拆解

**MCP解决的核心痛点——M×N问题：**

```
没有MCP：
  应用A → 适配GitHub API
  应用A → 适配数据库
  应用A → 适配文件系统
  应用B → 适配GitHub API（重复！）
  应用B → 适配数据库（重复！）
  ...
  M个应用 × N个工具 = M×N套代码

有MCP：
  应用A ─┐
  应用B ─┤→ MCP协议 → GitHub MCP Server
  应用C ─┘          → DB MCP Server
                     → Filesystem MCP Server
  M个应用 + N个Server = M+N套代码
```

**MCP协议的核心能力：**

| 能力 | 说明 | 类比 |
|:---:|------|------|
| **Tools（工具）** | 可执行的函数，如「发送邮件」「查询数据库」 | 按钮 |
| **Resources（资源）** | 可读取的数据，如「项目文件」「数据库表」 | 文件柜 |
| **Prompts（提示模板）** | 预定义的Prompt模板，如「代码审查模板」 | 模板库 |

**MCP的工作流程：**

```
① Client连接Server → 建立通信通道
② Client获取能力列表 → "你有哪些Tools/Resources/Prompts？"
③ Server返回清单 → 模型看到可用能力
④ 用户提问 → 模型决定调用某个Tool
⑤ Client转发调用请求 → Server执行真实操作
⑥ Server返回结果 → 模型根据结果继续回答
```

### 面试话术

> "MCP是Anthropic提出的模型上下文协议，解决的是AI应用对接外部工具时的M×N问题。没有MCP时，M个应用要对接N个工具需要写M×N套适配代码；MCP把它简化成M+N——应用实现一次Client，工具实现一次Server，通过标准协议通信。
>
> MCP提供三种能力：Tools（可执行函数）、Resources（可读数据）、Prompts（提示模板）。它和Function Calling不冲突，MCP负责工具的发现和接入，Function Calling负责模型决定调哪个工具。可以理解为MCP是工具的USB接口标准，FC是模型大脑的决策能力。"

### 雷区雷达

| 雷区 | 说明 |
|:---:|------|
| ❌ "MCP替代了Function Calling" | 两者互补，不替代 |
| ❌ "MCP是一个框架/库" | MCP是协议，不是具体实现 |
| ⚠️ 混淆Tools和Resources | Tools是执行操作，Resources是读取数据 |

## Q5：MCP由哪几部分组成？

### 一句话破题

MCP架构由三个角色组成：Host（宿主应用）→ Client（客户端）→ Server（服务端），形成「应用层→协议层→工具层」的三层结构。

### 类比记忆

> 🎯 **类比：餐厅运营**
>
> - **Host = 餐厅大堂经理** — 接待客人（用户），管理整个餐厅运转
> - **Client = 服务员** — 负责和后厨沟通，传递点单和上菜
> - **Server = 后厨** — 真正做菜的地方，每家后厨专做一类菜（GitHub Server做代码菜、DB Server做数据菜）
>
> 一个大堂经理可以管多个服务员，每个服务员对接一个后厨。

### 深度拆解

**三层架构详解：**

```
┌─────────────────────────────────┐
│           Host（宿主应用）         │
│  如：Claude Desktop / Cursor     │
│  职责：管理用户交互、协调多个Client  │
└──────────┬──────────────────────┘
           │ 管理
     ┌─────┴──────┐
     ↓            ↓
┌─────────┐  ┌─────────┐
│ Client A │  │ Client B │   ← 每个Client对应一个Server
│（协议层） │  │（协议层） │      负责通信协商、能力发现
└────┬────┘  └────┬────┘
     │            │
     ↓            ↓
┌─────────┐  ┌─────────┐
│ GitHub   │  │ Database │   ← Server实现具体工具能力
│ Server   │  │ Server   │      暴露Tools/Resources/Prompts
└─────────┘  └─────────┘
```

| 角色 | 职责 | 举例 |
|:---:|------|------|
| **Host** | 宿主应用，用户直接交互的界面，管理多个Client的生命周期 | Claude Desktop、Cursor、自研Agent应用 |
| **Client** | 协议通信层，每个Client与一个Server一对一连接，负责能力发现和消息转发 | Host内部自动创建，开发者通常不直接操作 |
| **Server** | 工具提供方，实现具体能力（Tools/Resources/Prompts） | GitHub MCP Server、Filesystem MCP Server |

**通信协议细节：**

| 传输方式 | 特点 | 适用场景 |
|:---:|------|------|
| **stdio** | 标准输入输出，本地进程通信 | 本地开发、Desktop应用 |
| **Streamable HTTP** | 基于HTTP，支持SSE流式推送 | 远程Server、生产部署 |

**Server暴露能力的三种原语：**

```python
# 1. Tool — 可执行的函数
@mcp.tool()
def search_code(query: str) -> str:
    """在代码库中搜索"""
    return codebase.search(query)

# 2. Resource — 可读取的数据源
@mcp.resource("file://project/{path}")
def read_file(path: str) -> str:
    """读取项目文件"""
    return open(path).read()

# 3. Prompt — 预定义的提示模板
@mcp.prompt()
def code_review(code: str) -> str:
    """代码审查模板"""
    return f"请审查以下代码：\n{code}"
```

### 面试话术

> "MCP架构分三层：Host是宿主应用，比如Claude Desktop或Cursor，负责用户交互和管理多个Client。Client是协议通信层，每个Client和Server一对一连接，负责能力发现和消息转发。Server是工具提供方，实现具体的Tools、Resources和Prompts三种能力。
>
> 通信方式上，本地开发用stdio，远程部署用Streamable HTTP。一个Host可以同时连接多个Server，比如同时连GitHub Server和数据库Server，模型可以跨工具协同工作。"

## Q6：MCP和Function Calling有什么区别？

### 一句话破题

Function Calling是模型层面的能力（决定调什么、传什么参数），MCP是协议层面的标准（工具怎么被发现、怎么被接入）——一个管「大脑决策」，一个管「接口规范」。

### 核心三问

**Q：两者是替代关系吗？**
不是。MCP的Server暴露工具能力后，模型仍然用Function Calling来决策调用哪个工具。MCP是「工具发现和接入层」，FC是「模型决策层」。

**Q：能不能只用FC不用MCP？**
能。小规模应用直接在代码里硬编码函数描述就行。MCP的价值在工具多、应用多的大规模场景。

**Q：能不能只用MCP不用FC？**
不能。MCP只负责把工具能力告诉模型，模型怎么决策调哪个工具还是要靠FC（或类似的机制）。

### 类比记忆

> 🎯 **类比：餐厅点餐**
>
> - **MCP = 菜单标准** — 规定菜单怎么写（统一格式：菜名、描述、价格、配料）
> - **Function Calling = 你点菜的大脑** — 看完菜单后决定点什么菜
>
> 菜单标准让所有餐厅的菜单格式统一（你不用学新格式），但点什么菜还是你的大脑决定的。两者一个管格式，一个管决策。

### 深度拆解

**对比全景：**

| 维度 | Function Calling | MCP |
|:---:|:---:|:---:|
| **本质** | 模型能力 | 通信协议 |
| **层级** | 模型决策层 | 工具接入层 |
| **解决的问题** | 模型怎么选工具、填参数 | 工具怎么被发现、被标准化接入 |
| **谁是主体** | LLM（OpenAI/Anthropic等） | 应用和工具之间的通信 |
| **输入** | 函数描述（JSON） | Tools/Resources/Prompts |
| **输出** | 函数调用JSON | 工具执行结果 |
| **标准化** | 各厂商格式不同 | 开放协议，跨厂商通用 |
| **动态发现** | ❌ 需要硬编码函数列表 | ✅ 运行时动态发现工具能力 |

**两者协作的完整流程：**

```
① MCP Client连接Server → 获取可用Tools列表
② Tools列表转换为FC格式 → 作为函数描述传给模型
③ 用户提问 → 模型用FC能力决策调哪个Tool
④ FC输出调用JSON → MCP Client转发给Server
⑤ Server执行 → 返回结果
⑥ 结果送回模型 → 生成最终回答
```

> 💡 **一句话总结：MCP负责「把工具送到模型面前」，FC负责「模型决定用哪个工具」。**

### 面试话术

> "MCP和Function Calling不是替代关系，而是互补关系。FC是模型层面的能力，解决的是'模型看到工具描述后怎么决策调用'的问题。MCP是协议层面的标准，解决的是'工具怎么被标准化发现和接入'的问题。
>
> 实际运行中两者配合使用：MCP Client从Server动态获取工具列表，转换为FC格式传给模型，模型用FC能力决策调哪个工具，MCP再负责转发执行。MCP的动态发现能力是FC不具备的——FC需要在代码里硬编码函数列表，MCP可以运行时动态获取。"

### 雷区雷达

| 雷区 | 说明 |
|:---:|------|
| ❌ "MCP替代了FC" | 两者互补，MCP+FC一起用 |
| ❌ "MCP是FC的升级版" | 不是一个层面的东西 |
| ⚠️ 忽略动态发现 | FC的工具列表是硬编码的，MCP是动态发现的，这是关键区别 |

## Q7：什么场景用Function Calling，什么场景用MCP？

### 一句话破题

**小规模、固定工具 → FC够用；大规模、多工具、跨应用 → 上MCP。** 选型核心看工具数量和复用需求。

### 深度拆解

**选型决策树：**

```
工具数量 < 5 且 只有1个应用用？
  └─ YES → Function Calling，直接硬编码
  └─ NO
      ├─ 需要跨多个应用复用同一套工具？
      │    └─ YES → MCP，写一次Server多应用共享
      │    └─ NO → FC也行，但注意维护成本
      └─ 需要动态发现/热插拔工具？
           └─ YES → MCP
           └─ NO → FC够用
```

**场景对比：**

| 场景 | 推荐 | 原因 |
|:---:|:---:|------|
| 内部Agent，3-5个固定工具 | FC | 工具少且固定，MCP引入额外复杂度不值得 |
| IDE插件（Cursor/VS Code），对接多个外部服务 | MCP | 工具多且可插拔，用户按需启用 |
| 多个Agent应用共享同一套工具 | MCP | 写一次Server，所有应用复用 |
| 快速原型/Demo | FC | 几行代码搞定，快速验证 |
| 生产级AI平台，支持第三方工具生态 | MCP | 标准化接入，生态可扩展 |
| 只用OpenAI API的单体应用 | FC | 原生支持，无需额外组件 |

### 面试话术

> "选型核心看工具规模和复用需求。如果工具少（5个以内）、只有单个应用使用，FC直接硬编码就够了，简单高效。如果工具多、需要跨应用复用、或者需要动态发现和热插拔，就上MCP——写一次Server所有应用共享，还能运行时动态获取工具列表。
>
> 实际项目里两者经常配合：底层用MCP做工具接入和发现，上层模型仍然用FC做决策。不是二选一的关系。"

## Q8：为什么有些推理模型不支持MCP协议？

### 一句话破题

推理模型（如o1/o3系列）的核心设计目标是「慢思考深度推理」，而工具调用需要「快决策即时响应」，两种模式在训练目标和推理流程上存在冲突。

### 核心三问

**Q：技术上看，推理模型不能调函数吗？**
技术上可以，但推理模型在「思考阶段」输出的是隐藏的思维链，不直接暴露给外部系统，无法在思考中途插入函数调用。

**Q：为什么不设计成支持？**
推理模型的训练目标是「在内部完成多步推理」，引入工具调用会打断推理链。而且推理模型的输出格式是结构化的推理过程，不是直接的函数调用JSON。

**Q：有什么变通方案？**
用「推理模型做规划 + 普通模型做执行」的两阶段架构：推理模型分析问题、制定计划，把计划传给支持FC的模型去执行工具调用。

### 类比记忆

> 🎯 **类比：数学家 vs 实验员**
>
> - 推理模型 = 数学家 — 擅长在脑子里推导证明，不擅长中途停下来去做实验
> - 普通模型 = 实验员 — 可以随时停下来用仪器测量、查资料
>
> 让数学家推导到一半停下来去做实验，会打断他的思路。更好的方式是数学家先把推导思路写出来，交给实验员去执行。

### 深度拆解

**推理模型 vs 普通模型在工具调用上的差异：**

| 维度 | 推理模型（o1/o3） | 普通模型（GPT-4o） |
|:---:|:---:|:---:|
| 核心能力 | 深度推理、多步思考 | 快速响应、工具调用 |
| 输出格式 | 隐藏思维链 + 最终答案 | 直接文本/函数调用JSON |
| 推理过程 | 内部多步推理，不可中断 | 逐步生成，可随时插入工具调用 |
| 训练目标 | 推理准确性 | 指令遵循+工具调用 |
| 响应延迟 | 高（思考时间长） | 低 |

**变通方案——两阶段架构：**

```
阶段1: 推理模型做规划
  用户问题 → o1分析 → 输出执行计划
  "要回答这个问题，需要：
   1. 调用search_api查找相关资料
   2. 调用calculator做数值计算
   3. 综合结果生成答案"

阶段2: 普通模型做执行
  执行计划 → GPT-4o按计划调用工具 → 返回结果
```

### 面试话术

> "推理模型不支持MCP主要因为训练目标和推理流程的冲突。推理模型的核心是内部多步深度推理，思维链是隐藏的不对外暴露，无法在推理中途插入工具调用。而且推理模型的训练目标是推理准确性，没有专门训练工具调用能力。
>
> 变通方案是用两阶段架构：推理模型负责分析问题和制定计划，把计划交给支持FC的普通模型去执行工具调用。这样各取所长。"

## Q9：Skill是什么？

### 一句话破题

Skill是Agent的「能力包」——一组预定义的Prompt模板 + 工具配置 + 执行流程，让Agent在面对特定任务时自动加载对应的能力组合，而不需要每次都从零开始推理。

### 类比记忆

> 🎯 **类比：职业技能证书**
>
> 一个电工上岗前要考取「电工证」，证书代表他具备了电工相关的一整套知识和操作规范。Agent的Skill就是它的「职业证书」——拿到「代码审查Skill」就自动具备了代码审查的Prompt模板、相关工具和执行流程。
>
> 没有Skill的Agent = 什么都会一点但什么都不精的通才
> 有Skill的Agent = 针对特定任务经过专业训练的专家

### 深度拆解

**Skill的组成结构：**

```yaml
name: "code_review"
description: "审查代码质量和安全性"
trigger: "当用户请求代码审查时自动激活"

components:
  system_prompt: |
    你是一个专业的代码审查专家。
    审查要点：1.安全漏洞 2.性能问题 3.代码规范 4.可维护性

  tools:
    - read_file      # 读取代码文件
    - search_code    # 搜索代码库
    - run_linter     # 运行代码检查工具

  workflow:
    - step1: 读取待审查的代码
    - step2: 运行linter自动检查
    - step3: 逐行分析代码
    - step4: 生成审查报告
```

**Skill vs 普通Prompt的区别：**

| 维度 | 普通Prompt | Skill |
|:---:|:---:|:---:|
| 组成 | 纯文本指令 | Prompt + 工具 + 流程 + 触发条件 |
| 触发 | 手动粘贴 | 自动识别任务类型激活 |
| 工具 | 需要单独配置 | 内置工具配置 |
| 复用 | 每次重新写 | 打包复用，一键加载 |
| 复杂度 | 简单 | 结构化、可组合 |

### 面试话术

> "Skill是Agent的能力包，把特定任务所需的Prompt模板、工具配置和执行流程打包在一起。当Agent识别到特定任务类型时自动加载对应Skill，不需要每次从零推理。
>
> 和普通Prompt的区别在于：Skill是结构化的能力单元，包含触发条件、工具配置和执行流程，可以自动激活和复用。类似于给Agent发了一张职业证书，拿到证书就自动具备了对应的专业能力。"

## Q10：MCP和Agent Skill的区别是什么？

### 一句话破题

MCP提供「工具接入能力」（让Agent能用工具），Skill提供「任务执行能力」（让Agent知道怎么用工具完成任务）——一个管「有什么工具」，一个管「怎么用工具」。

### 类比记忆

> 🎯 **类比：厨房运营**
>
> - **MCP = 食材供应链** — 确保厨房有各种食材和厨具可用（提供原材料和工具）
> - **Skill = 菜谱** — 告诉厨师做某道菜的完整步骤（提供怎么做的方法论）
>
> 有食材没菜谱 → 厨师不知道做什么菜
> 有菜谱没食材 → 巧妇难为无米之炊
> 两者配合 → 厨师看到点菜单，选对应菜谱，用食材做菜

### 深度拆解

| 维度 | MCP | Agent Skill |
|:---:|:---:|:---:|
| **本质** | 通信协议 | 能力封装 |
| **提供什么** | 工具的发现和接入通道 | 完整的任务执行方案 |
| **包含内容** | Tools/Resources/Prompts | Prompt + 工具 + 流程 + 触发条件 |
| **关注点** | 「有什么工具可用」 | 「怎么用工具完成任务」 |
| **粒度** | 单个工具级别 | 任务级别 |
| **动态性** | 运行时动态发现 | 预定义，按需加载 |
| **比喻** | USB接口标准 | 应用程序 |

**协作关系：**

```
用户: "帮我审查这段代码"
  ↓
Agent识别任务 → 激活 "code_review" Skill
  ↓
Skill定义了流程:
  1. 用MCP的read_file工具读代码
  2. 用MCP的search_code工具搜索相关代码
  3. 用MCP的run_linter工具检查
  4. 综合分析生成报告
  ↓
MCP提供工具执行通道 → 返回结果
  ↓
Skill按流程整合结果 → 输出审查报告
```

### 面试话术

> "MCP和Skill不在一个层面。MCP是协议层，解决工具的发现和接入——让Agent知道有哪些工具可用。Skill是能力层，解决任务的执行方案——告诉Agent面对特定任务时该怎么组合使用这些工具。
>
> 类比来说，MCP是食材供应链确保厨房有食材，Skill是菜谱告诉厨师怎么做菜。实际运行中Skill调用MCP提供的工具来完成任务，两者是互补关系。"

## Q11：Function Calling、Skill、MCP三者有什么区别？

### 一句话破题

FC是**模型能力**（决定调什么），MCP是**接入协议**（工具怎么连），Skill是**任务方案**（怎么组合工具完成任务）——三个不同层面的东西，可以同时使用。

### 类比记忆

> 🎯 **类比：开一家餐厅**
>
> | 概念 | 类比 | 解决的问题 |

> | Skill | 菜谱 | 每道菜的完整制作流程和配方 |
>
> 厨师大脑（FC）看菜谱（Skill），用供应链（MCP）提供的食材做菜。

### 深度拆解

**三层关系全景图：**

```
┌──────────────────────────────────────────┐
│              Skill（任务方案层）             │
│  "代码审查Skill": read_file→search→lint→报告 │

│  模型根据Skill的流程，决策每步调哪个工具       │
├──────────────────────────────────────────┤
│              MCP（工具接入层）               │
│  提供read_file/search_code/run_linter工具    │
└──────────────────────────────────────────┘
```

**三者对比矩阵：**

| 维度 | Function Calling | MCP | Skill |
|:---:|:---:|:---:|:---:|
| 层级 | 模型决策层 | 协议接入层 | 任务方案层 |
| 本质 | 模型能力 | 通信协议 | 能力封装 |

>
> 三者可以同时使用：Skill定义任务流程，FC在每一步决策调什么工具，MCP提供工具的执行通道。不是三选一，而是从上到下的三层架构。"

## Q12：什么是A2A协议？它和MCP的区别是什么？

### 一句话破题

A2A（Agent-to-Agent）协议解决的是「Agent和Agent之间怎么协作」的问题，MCP解决的是「Agent和工具之间怎么连接」的问题——一个管Agent间通信，一个管Agent与工具通信。

### 类比记忆

> 🎯 **类比：公司运营**
>
> - **MCP = 员工和工具的关系** — 员工（Agent）怎么使用办公系统、数据库等工具
> - **A2A = 员工之间的关系** — 员工A怎么把任务交接给员工B，怎么汇报进度，怎么协调资源
>
> MCP让单个Agent能力更强（有工具用），A2A让多个Agent协同作战（能分工合作）。

### 深度拆解

**A2A vs MCP 对比：**

| 维度 | MCP | A2A |
|:---:|:---:|:---:|
| 通信双方 | Agent ↔ 工具 | Agent ↔ Agent |
| 解决问题 | 工具接入标准化 | Agent协作标准化 |
| 交互模式 | 请求-响应 | 委托-执行-汇报 |
| 关系 | 主从（Agent控制工具） | 平等（Agent之间协商） |
| 典型场景 | Agent调用数据库/文件/API | 多个Agent分工合作完成大任务 |

**A2A的核心机制：**

```
场景：用户要求"分析竞品并生成报告"

① 协调Agent收到任务 → 分解为子任务
② 协调Agent通过A2A协议委托:
   → 数据采集Agent: "收集竞品A的价格数据"
   → 分析Agent: "对数据进行SWOT分析"
   → 写作Agent: "将分析结果写成报告"
③ 各Agent执行各自任务 → 通过A2A汇报结果
④ 协调Agent汇总 → 返回给用户
```

**A2A协议的核心要素：**

| 要素 | 说明 |
|:---:|------|
| Agent Card | Agent的「名片」，描述自己的能力 |
| Task | 可委托的任务，包含描述、状态、结果 |
| Message | Agent间的通信消息 |
| Artifact | 任务产出的成果物 |

### 面试话术

> "A2A是Agent-to-Agent的协作协议，解决的是多个Agent之间怎么分工合作的问题。MCP解决的是单个Agent怎么接入外部工具的问题。
>
> 类比来说，MCP是员工和办公工具的关系，A2A是员工之间的关系。在多Agent系统中两者配合使用：每个Agent通过MCP连接自己的工具集，Agent之间通过A2A协议协调任务分工和结果汇报。"

## Q13：MCP协议通常采用什么通信方式？

### 一句话破题

MCP支持两种传输方式：**stdio**（本地进程通信）和 **Streamable HTTP**（远程网络通信，内部可用SSE做流式推送）。

### 深度拆解

| 传输方式 | 原理 | 适用场景 | 特点 |
|:---:|------|:---:|------|
| **stdio** | 通过标准输入/输出流通信 | 本地开发、Desktop应用 | 简单、零配置、低延迟 |
| **Streamable HTTP** | 基于HTTP协议，支持SSE流式推送 | 远程Server、生产部署 | 可跨网络、支持鉴权、可流式 |

**stdio模式工作原理：**

```
Host应用
  ↓ 启动子进程 (MCP Server)
  ↓ 通过stdin发送JSON-RPC请求
  ↑ 通过stdout接收JSON-RPC响应
```

**Streamable HTTP模式工作原理：**

```
Client → POST /mcp (JSON-RPC请求)
Client ← SSE流式响应 (逐步推送结果)
  或
Client ← 普通HTTP响应 (一次性返回)
```

> ⚠️ **注意：** MCP早期规范（2024-11）使用SSE+HTTP双端点模式，2025-03规范升级为**单端点Streamable HTTP**，简化了架构，同时保持了代理穿透友好性。

### 面试话术

> "MCP支持两种传输方式。本地场景用stdio——Host启动Server子进程，通过标准输入输出通信，零配置低延迟。远程场景用Streamable HTTP——基于HTTP协议，支持SSE做流式推送。2025年3月的规范从早期的SSE+HTTP双端点简化成了单端点模式，降低了实现复杂度。"

## Q14：说说WebSocket和SSE通信的区别及局限性？

### 一句话破题

**SSE是服务端单向推送（水管），WebSocket是全双工双向通信（电话）**——LLM文字对话绝大多数用SSE就够了，只有在需要用户实时打断等双向场景才上WebSocket。

### 核心三问

**Q：为什么OpenAI和Anthropic都用SSE而不是WebSocket？**
LLM流式输出是「服务端单向推token，客户端只看」的场景，SSE完全够用，而且轻量、HTTP原生支持、代理穿透好。

**Q：SSE的局限性是什么？**
单向（客户端发消息要另起HTTP请求）、HTTP/1.1连接数上限（6条）、只能传文本（二进制要Base64编码膨胀33%）。

**Q：WebSocket的局限性是什么？**
有状态导致横向扩展困难（需Redis做连接状态外移）、容易被企业代理/防火墙拦截、没有内置请求-响应配对机制。

### 类比记忆

> 🎯 **类比：沟通方式**
>
> - **标准HTTP** = 信件 — 一问一答，有来有回，但等回信时干不了别的
> - **SSE** = 广播 — 电台单向播，你只能听，想说话得另打电话
> - **WebSocket** = 电话 — 双方随时开口，真正双向实时
>
> 听广播（SSE）够用时，没必要打电话（WebSocket）。

### 深度拆解

**技术对比全景：**

| 维度 | SSE | WebSocket |
|:---:|:---:|:---:|
| 通信方向 | 服务端→客户端（单向） | 全双工（双向） |
| 底层协议 | HTTP/1.1（巧用） | 独立协议（基于TCP） |
| 建立方式 | 普通HTTP GET + Accept头 | HTTP Upgrade握手 → 101 |
| 数据格式 | 纯文本 | 文本+二进制 |
| 断线重连 | 内置（EventSource自动重连） | 需自己实现 |
| 代理穿透 | ✅ 任何代理都能透传 | ❌ 可能被拦截 |
| 横向扩展 | ✅ 简单（无状态HTTP） | ❌ 困难（有状态连接） |
| 连接数限制 | HTTP/1.1同域名6条 | 无此限制 |
| 浏览器支持 | EventSource API | WebSocket API |

**SSE在LLM场景的优势：**

```
用户提问 → POST /chat (发送问题)
        ← SSE流 (逐token推送回答)
           data: {"token":"你"}
           data: {"token":"好"}
           data: {"token":"！"}
           data: [DONE]
```

**什么时候该用WebSocket：**
- 用户需要在模型输出中途打断 → 需要客户端在流中途发消息
- 多人协同实时编辑 → 频繁双向消息
- 实时语音对话 → 但实际更推荐WebRTC

### 面试话术

> "SSE和WebSocket最本质的区别是通信方向：SSE是服务端单向推送，客户端发消息要另起HTTP请求；WebSocket是全双工双向通信。
>
> 在LLM场景下，流式文字输出是单向推送，SSE完全够用，这也是OpenAI和Anthropic都用SSE的原因。SSE的优势是轻量、HTTP原生支持、代理穿透好、横向扩展简单。WebSocket的复杂性只有在需要双向实时通信（如中途打断、多人协同）时才值得引入。
>
> SSE的局限是HTTP/1.1连接数上限6条、只能传文本、单向导致双通道架构。WebSocket的局限是有状态导致扩展困难、容易被代理拦截。"

### 雷区雷达

| 雷区 | 说明 |
|:---:|------|
| ❌ "WebSocket更强大所以更好" | 杀鸡用牛刀，LLM文字对话SSE够用 |
| ❌ "SSE是WebSocket的简化版" | 两者是不同协议，不是简化关系 |
| ⚠️ 忽略代理穿透 | 企业环境WebSocket可能被防火墙拦截，SSE无此问题 |

## Q15：为什么用WebRTC？它和WebSocket在AI对话流中的核心差异是什么？

### 一句话破题

WebRTC基于UDP实现低延迟实时音视频传输，而WebSocket基于TCP——在实时语音对话场景，TCP的重传机制反而成了延迟瓶颈，必须用UDP才能做到「丢一个包无所谓，下一个包赶紧到」。

### 核心三问

**Q：为什么TCP不适合实时语音？**
TCP保证可靠有序传输，丢包会重传。语音场景中，一个100ms前的音频包重传到了也没用（已经过了播放时间），反而造成延迟堆积。UDP丢了就丢了，不影响后续包。

**Q：WebRTC只是传输协议吗？**
不只是传输协议，WebRTC是一整套实时通信方案，包含音视频编解码、NAT穿透、网络自适应等，浏览器原生支持。

**Q：WebSocket不能做语音对话吗？**
能做但不理想。WebSocket基于TCP，TCP的队头阻塞和重传机制在弱网下会导致延迟飙升。而且WebSocket传音频要先Base64编码，数据量膨胀33%。

### 类比记忆

> 🎯 **类比：寄包裹**
>
> - **TCP（WebSocket底层）** = 挂号信 — 丢了必须补寄，保证每封都到，但慢
> - **UDP（WebRTC底层）** = 普通快递 — 丢了就算了，下一个赶紧送，快但不保证到
>
> 文字聊天用挂号信（TCP），丢一个字整句话都变了，必须保证到。语音通话用普通快递（UDP），丢100ms音频无所谓，但不能因为补寄导致后面的声音都卡住。

### 深度拆解

**三种协议在AI对话中的定位：**

| 协议 | 底层 | 适用场景 | 延迟特性 |
|:---:|:---:|:---:|------|
| SSE | TCP | 文字流式输出 | 可接受几百ms延迟 |
| WebSocket | TCP | 双向文字交互 | 可接受几百ms延迟 |
| WebRTC | UDP | 实时语音/视频对话 | 要求<200ms端到端 |

**为什么语音必须用UDP：**

```
语音包序列: [包1] [包2] [包3] [包4] [包5]

TCP场景（包3丢了）:
  [包1] [包2] [等待包3重传...] [包4] [包5]
  → 包4和包5被阻塞，用户听到: "你好...（卡顿）...世界"
  → 延迟越来越堆积

UDP场景（包3丢了）:
  [包1] [包2] [×] [包4] [包5]
  → 包4和包5正常到达，用户听到: "你好（轻微杂音）世界"
  → 延迟不堆积，体验流畅
```

**WebRTC在AI语音对话中的完整能力：**

| 能力 | 说明 |
|:---:|------|
| UDP传输 | 低延迟，丢包不阻塞 |
| 音频编解码 | Opus编解码器，低码率高质量 |
| NAT穿透 | STUN/TURN服务器，穿透防火墙 |
| 回声消除 | 浏览器内置AEC，防止扬声器声音回传 |
| 自适应码率 | 网络差时自动降低音质保流畅 |

### 面试话术

> "WebRTC和WebSocket的核心差异在底层传输协议：WebRTC基于UDP，WebSocket基于TCP。在实时语音对话场景，TCP的可靠重传机制反而是负担——语音包丢了重传到了已经过了播放时间，还会阻塞后续包导致延迟堆积。UDP丢了就丢了，不影响后续传输。
>
> 此外WebRTC是一整套实时通信方案，包含音频编解码、NAT穿透、回声消除等能力，浏览器原生支持。文字对话用SSE/WebSocket，实时语音/视频用WebRTC，这是业界共识。"

## Q16：有没有用过大模型的网关框架？网关层解决了什么问题？

### 一句话破题

LLM网关是架在应用和模型API之间的中间层，解决的不只是负载均衡，更核心的是API Key集中管理、按团队token配额、成本追踪、语义缓存和Prompt安全过滤——这些都是普通API网关做不了的。

### 核心三问

**Q：LLM网关和普通API网关有什么本质区别？**
普通API网关只做路由和限流。LLM网关额外处理token级别的计量、语义缓存（不是精确匹配而是向量相似度匹配）、Prompt安全过滤等LLM特有问题。

**Q：语义缓存是怎么工作的？**
把用户问题转成向量，在向量库里做相似度搜索，找到语义相近的历史问题就直接返回历史答案，完全跳过LLM调用。不是精确匹配，是「意思接近就命中」。

**Q：用过什么网关框架？**
LiteLLM（最活跃的开源LLM网关，支持100+模型）、One API（国内活跃，支持国产模型）、PortKey（商业+开源）。

### 类比记忆

> 🎯 **类比：企业前台**
>
> 没有前台：每个员工自己点外卖、自己收快递、自己接待客人 → 乱
>
> 有前台（网关）：
> - 统一收发快递（多模型统一接口）
> - 管理门禁卡（API Key集中管理）
> - 部门预算控制（token配额）
> - 记录每月快递费用（成本追踪）
> - 智能记忆：上次有人问过同样的问题直接回答（语义缓存）
> - 安全检查：可疑包裹先拆检（Prompt安全过滤）

### 深度拆解

**没有网关时的五大痛点：**

| 痛点 | 场景 | 后果 |
|:---:|------|------|
| API Key散落 | 每个服务各自存Key | 一处泄露=安全事故，某工程师离职后Key未清理被黑客扫到 |
| 重复劳动 | 每个服务自己写重试/限流 | 逻辑不一致，出bug难排查 |
| 成本黑箱 | 不知道哪个服务烧多少钱 | 月底3万美元账单无法分摊 |
| 灵活性差 | 换模型要改代码 | 想从GPT-4o换Claude要改所有调用代码 |
| 无配额控制 | 一个团队跑死循环脚本 | 一晚上吃光全周token预算 |

**LLM网关的六大核心能力：**

```
                        ┌─ ① 多模型统一接口（OpenAI兼容格式）
                        ├─ ② API Key集中管理（业务拿虚拟Key）
应用 →→→ [ LLM网关 ] ──├─ ③ 按团队token配额（独立预算）
                        ├─ ④ 成本追踪（token级计量）
                        ├─ ⑤ 语义缓存（向量相似度匹配）
                        └─ ⑥ Prompt安全过滤（注入检测/PII过滤）
                              ↓
                    OpenAI / Anthropic / 其他API
```

**语义缓存详解（LLM网关的亮点功能）：**

```
① 收到新问题: "北京今天热吗"
② 把问题转成向量embedding
③ 在向量数据库搜索相似历史问题
④ 找到: "北京现在天气怎样"（相似度0.92 > 阈值0.85）
⑤ 直接返回历史答案，跳过LLM调用！
⑥ 省钱 + 降延迟
```

关键调参：
- 相似度阈值：0.85-0.95之间，太高缓存形同虚设，太低容易答非所问
- 缓存时效：实时信息缓存几分钟，FAQ缓存几天

**常见网关框架：**

| 框架 | 特点 | 适合场景 |
|:---:|------|------|
| LiteLLM | 开源Python，100+模型，社区最活跃 | 通用场景 |
| One API | 开源Go，国内活跃，支持国产模型 | 国内项目 |
| Bifrost | 开源Rust，高性能低延迟 | 性能敏感场景 |
| PortKey | 商业+开源，功能完整 | 不想自运维的团队 |

### 面试话术

> "我用过LiteLLM。LLM网关解决的不只是负载均衡，更核心的是五个LLM特有的问题：
>
> 第一，API Key集中管理，业务服务拿虚拟Key，接触不到真实密钥，降低泄露风险。第二，按团队做token配额，防止某个团队把整个公司的额度用光。第三，成本追踪，所有调用的token用量集中记录，知道哪个服务最烧钱。第四，语义缓存——这是LLM网关区别于普通API网关的亮点，用向量相似度匹配语义相近的问题，命中缓存直接返回历史答案跳过LLM调用。第五，Prompt安全过滤，统一检测注入攻击和隐私信息。
>
> 换模型时只需改网关路由配置，业务代码一行不动。故障转移也是网关层自动处理的，主路由连续失败自动切备用路由。"

### 雷区雷达

| 雷区 | 说明 |
|:---:|------|
| ❌ "网关就是做负载均衡的" | 负载均衡只是基础功能，LLM特有能力才是核心价值 |
| ❌ "Nginx反向代理就够了" | Nginx做不了token计量、语义缓存、Prompt安全 |
| ⚠️ 忽略语义缓存调参 | 阈值设错要么缓存没用要么答非所问 |
| ⚠️ LiteLLM安全提醒 | 2026年3月有过供应链安全事件，生产使用需关注安全公告 |
