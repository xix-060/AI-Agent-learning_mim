## Q1：什么是大语言模型？和传统NLP模型有什么区别？

### 一句话破题

大语言模型（LLM）是一个通过海量文本训练的、能理解和生成自然语言的超大规模神经网络——核心区别在于「通用性」：传统NLP模型一个任务一个模型，LLM一个模型搞定所有NLP任务。

### 核心三问

**Q：为什么叫"大"语言模型？**
参数量大（十亿到万亿级）、训练数据量大（TB级文本）、计算资源大（千卡集群训练）。"大"是这些规模的统称。

**Q：和传统NLP模型的本质区别是什么？**
传统NLP = 特征工程 + 任务专属小模型（一个情感分析模型只能做情感分析）。LLM = 预训练+微调范式，一个模型通过Prompt适配无数任务。

**Q：LLM为什么能做到"通用"？**
预训练阶段学习了语言的统计规律和世界知识，这些知识是任务无关的。通过Prompt指定任务，同一个模型可以切换做翻译、摘要、问答、代码生成等。

### 类比记忆

> 🎯 **类比：瑞士军刀 vs 专用工具**
>
> - **传统NLP模型** = 一把螺丝刀 — 只能拧螺丝，干不了别的
> - **大语言模型** = 瑞士军刀 — 一把工具集成了刀、剪、锯、开瓶器，按需使用
>
> 传统方式做一个NLP应用要招5个工程师训练5个模型，LLM时代一个模型+不同的Prompt就搞定了。

### 深度拆解

**范式对比：**

| 维度 | 传统NLP | 大语言模型 |
|:---:|:---:|:---:|
| 模型架构 | RNN/CNN/小Transformer | 大规模Transformer |
| 参数量 | 百万~亿级 | 十亿~万亿级 |
| 训练方式 | 任务专属训练 | 预训练+微调/对齐 |
| 任务适配 | 换任务要重新训练 | 换Prompt即可 |
| 数据需求 | 标注数据（每任务几万条） | 预训练用海量无标注文本 |
| 泛化能力 | 弱（只擅长训练过的任务） | 强（zero-shot/few-shot） |
| 典型代表 | BERT-base、TextCNN | GPT-4、Claude、Qwen |

**LLM的核心能力来源：**

```
海量文本预训练 → 学会语言规律 + 世界知识
       ↓
指令微调(SFT) → 学会遵循人类指令
       ↓
偏好对齐(RLHF) → 输出更安全、有用、无害
       ↓
通用AI助手 — 能翻译、能写代码、能问答、能推理...
```

### 面试话术

> "大语言模型是通过海量文本预训练的超大规模Transformer模型。和传统NLP模型的本质区别是通用性——传统NLP是'一个任务一个模型'，LLM是'一个模型适配所有任务'，通过Prompt切换任务类型。
>
> 这种通用性来自三个阶段：预训练让模型学会语言规律和世界知识，SFT让模型学会遵循指令，RLHF让模型输出更符合人类偏好。参数规模从十亿到万亿级不等，'大'体现在参数量、数据量和计算量三个维度。"

## Q2：讲讲Transformer架构基本原理？Encoder和Decoder是什么？

### 一句话破题

Transformer是完全基于注意力机制的序列处理架构，核心是「让每个词都能直接看到序列中所有其他词」——Encoder负责理解输入，Decoder负责生成输出。

### 核心三问

**Q：Transformer为什么比RNN好？**
RNN只能顺序处理（看完第N个词才能看第N+1个），无法并行且长距离依赖差。Transformer用注意力机制让所有词同时交互，天然支持并行计算，且长距离依赖直接连接。

**Q：Encoder和Decoder的区别是什么？**
Encoder是双向的（每个词能看到前后所有词），适合理解任务。Decoder是单向的（每个词只能看到前面的词），适合生成任务。GPT系列只用Decoder，BERT只用Encoder。

**Q：自注意力机制的原理是什么？**
每个词生成三个向量Q（查询）、K（键）、V（值），用Q和所有词的K做点积算注意力权重，再用权重对V加权求和——本质是「当前词应该关注序列中哪些词」。

### 类比记忆

> 🎯 **类比：开会讨论**
>
> - **RNN** = 传话游戏 — 第一个人跟第二个人说，第二个跟第三个说...信息传递到后面早变形了
> - **Self-Attention** = 圆桌讨论 — 每个人都能直接听到所有人的发言，想关注谁就关注谁
> - **Encoder** = 会议纪要 — 综合所有人的发言理解整体内容
> - **Decoder** = 逐个发言 — 轮到你说话时只能参考之前说过的话，不能偷看后面的

### 深度拆解

**自注意力计算过程：**

```
输入序列: "我 爱 编程"

Step 1: 每个词生成Q、K、V三个向量
  我  → Q₁, K₁, V₁
  爱  → Q₂, K₂, V₂
  编程 → Q₃, K₃, V₃

Step 2: 计算注意力分数（Q·K）
  "我"对"我"的注意力: Q₁·K₁
  "我"对"爱"的注意力: Q₁·K₂
  "我"对"编程"的注意力: Q₁·K₃

Step 3: Softmax归一化 → 得到权重
  weights = softmax([Q₁·K₁, Q₁·K₂, Q₁·K₃])
  → 例如 [0.7, 0.2, 0.1]（"我"最关注自己）

Step 4: 加权求和V
  output₁ = 0.7×V₁ + 0.2×V₂ + 0.1×V₃

公式: Attention(Q,K,V) = softmax(Q·Kᵀ/√dₖ)·V
```

**Transformer完整架构：**

```
Encoder（×N层）:
  输入 → 自注意力 → Add&Norm → FFN → Add&Norm → 输出
         （双向）    （残差+   （前馈   （残差+
                     层归一）  网络）   层归一）

Decoder（×N层）:
  输入 → Masked自注意力 → Cross注意力 → FFN → 输出
         （单向，遮住未来）  （看Encoder输出）
```

**三大架构路线：**

| 路线 | 结构 | 代表模型 | 擅长 |
|:---:|:---:|:---:|:---:|
| Encoder-only | 双向理解 | BERT | 理解任务（分类、NER） |
| Decoder-only | 单向生成 | GPT、Qwen、LLaMA | 生成任务（对话、写作） |
| Encoder-Decoder | 理解+生成 | T5、BART | 翻译、摘要 |

> 💡 **趋势：** 当前主流大模型（GPT-4、Claude、Qwen等）几乎都是Decoder-only架构，因为生成能力最强且scaling law表现最好。

### 面试话术

> "Transformer的核心是自注意力机制，让序列中每个词都能直接和所有其他词交互，解决了RNN的顺序依赖和长距离衰减问题。
>
> 计算过程是：每个词生成Q、K、V三个向量，用Q和所有词的K做点积算注意力分数，softmax归一化后对V加权求和。多头注意力是并行做多个注意力，捕获不同维度的信息。
>
> Encoder是双向的，适合理解任务；Decoder是单向的（用Mask遮住未来token），适合生成任务。当前主流大模型都是Decoder-only架构，因为生成能力和scaling law表现最好。"

## Q3：多头注意力MHA有哪些局限？MQA、GQA、Flash Attention怎么解决？

### 一句话破题

MHA每个头有独立的K和V矩阵，KV Cache显存占用大；MQA所有头共享一组K/V，省显存但质量降；GQA分组共享折中两者；Flash Attention从计算层面优化GPU内存访问效率。

### 核心三问

**Q：MHA的局限是什么？**
每个注意力头都有独立的K和V矩阵，KV Cache显存随头数线性增长。7B模型32个头，长序列KV Cache可达17GB+，显存压力巨大。

**Q：MQA为什么省显存但掉点？**
所有头共享同一组K/V，KV Cache直接缩小到1/N（N=头数）。但所有头看到的信息完全一样，表达能力下降，模型质量明显降低。

**Q：GQA怎么折中的？**
把头分成G组，组内共享K/V，组间不同。比如32个头分8组，每组4个头共享K/V——显存降到1/4，质量接近MHA。

### 类比记忆

> 🎯 **类比：公司资料管理**
>
> - **MHA** = 每个部门各有完整资料库 — 资料全但占空间大
> - **MQA** = 全公司共用一个资料库 — 省空间但每个部门看到的信息一样，专业度降低
> - **GQA** = 按业务线分组共享资料库 — 每组4个部门共享一套资料，省空间又保持一定专业度
> - **Flash Attention** = 不改资料库结构，优化资料存取方式（从磁盘顺序读取变成内存块读取）

### 深度拆解

**四种方案对比：**

| 方案 | K/V矩阵数 | KV Cache大小 | 模型质量 | 代表模型 |
|:---:|:---:|:---:|:---:|:---:|
| MHA | N（=头数） | 1× | 最好 | GPT-3、BERT |
| MQA | 1 | 1/N | 明显下降 | PaLM、StarCoder |
| GQA | G（组数） | G/N | 接近MHA | LLaMA-2/3、Qwen2 |
| Flash Attention | 不变 | 不变 | 不变 | 优化计算方式，非架构改变 |

> 💡 **GQA已成为主流：** LLaMA-2/3、Qwen2等主流模型都采用GQA，在显存效率和质量之间取得最佳平衡。

**Flash Attention的核心原理：**

不是改变注意力机制，而是优化GPU内存访问模式：
- **问题：** 标准注意力要把N×N的注意力矩阵写入HBM（高带宽内存），再读回做softmax，IO开销巨大
- **解决：** 用分块计算（Tiling），在SRAM（片上高速缓存）里完成attention计算，避免中间结果写回HBM
- **效果：** 计算结果完全一致，但IO开销大幅降低，速度提升2-4倍

### 面试话术

> "MHA的局限是KV Cache显存随头数线性增长，7B模型长序列KV Cache可达17GB。MQA让所有头共享一组K/V，显存降到1/N但质量明显下降。GQA折中——把头分组，组内共享，LLaMA和Qwen等主流模型都用GQA。
>
> Flash Attention是另一维度的优化，不改注意力架构，而是优化GPU内存访问——标准注意力要把N×N矩阵写入HBM再读回，Flash Attention用分块计算在SRAM内完成，减少IO开销，速度提升2-4倍且结果完全一致。"

## Q4：大模型的位置编码是干什么用的？sin/cos、RoPE、ALiBi有什么区别？

### 一句话破题

位置编码给序列中每个词注入位置信息——因为自注意力机制本身是位置无关的（打乱词序结果不变），必须额外告诉模型"每个词在什么位置"。

### 核心三问

**Q：为什么自注意力需要位置编码？**
注意力计算只看Q·K的相似度，不关心位置。如果不加位置编码，"狗咬人"和"人咬狗"在注意力看来完全一样。

**Q：RoPE为什么成为主流？**
RoPE（旋转位置编码）通过旋转Q和K向量来编码相对位置，天然支持外推（训练时4K，推理时可扩展到更长），且不引入额外参数。LLaMA、Qwen都用RoPE。

**Q：ALiBi的特点是什么？**
ALiBi不用位置编码向量，而是在注意力分数上直接加一个距离惩罚（越远衰减越大）。优点是天然支持外推，缺点是表达能力有限。

### 类比记忆

> 🎯 **类比：排队买奶茶**
>
> 自注意力像一群人不排队乱站着讨论——每个人能听到所有人说话，但不知道谁在前面谁在后面。位置编码就是给每个人发一个号码牌——"你是1号，你是2号"，这样就能区分顺序了。
>
> - **sin/cos** = 固定号码牌 — 每个位置一个固定的编码
> - **RoPE** = 旋转号码牌 — 用旋转角度编码位置，能感知相对距离
> - **ALiBi** = 距离感号码牌 — 不发号码，而是让越远的人说话声音越小

### 深度拆解

**三种方案对比：**

| 方案 | 原理 | 外推能力 | 代表模型 |
|:---:|------|:---:|:---:|
| sin/cos | 固定的正弦/余弦函数生成位置向量 | 差 | 原始Transformer |
| RoPE | 旋转Q和K向量编码相对位置 | 好（配合缩放） | LLaMA、Qwen |
| ALiBi | 注意力分数加距离惩罚 | 好 | BLOOM、MPT |

**RoPE核心思想：**

```
位置m的向量 → 旋转m×θ角度
位置n的向量 → 旋转n×θ角度

Q·K的点积只依赖于角度差(m-n)
→ 天然编码了相对位置！
```

外推问题与解决：
- RoPE直接外推到训练长度之外会掉点
- 解决方案：YaRN、NTK-aware scaling等缩放方法，调整旋转基频

### 面试话术

> "位置编码解决的是自注意力机制位置无关的问题——注意力只看Q·K相似度，打乱词序结果不变，所以需要额外注入位置信息。
>
> 三种主流方案：sin/cos是固定编码，外推能力差；RoPE通过旋转Q/K向量编码相对位置，配合缩放可外推，是LLaMA、Qwen等主流模型的选择；ALiBi在注意力分数上直接加距离惩罚，天然支持外推但表达能力有限。
>
> RoPE的外推问题是当前研究热点，YaRN、NTK-aware等缩放方法可以把4K训练的模型扩展到32K+。"

## Q5：什么是大模型的分词器？原理是什么？

### 一句话破题

分词器（Tokenizer）是把文本切分成模型能处理的最小单元（Token）的工具——大模型不直接处理文字，而是处理Token ID（数字）。

### 核心三问

**Q：为什么不能按字切分？**
纯按字切分（中文每字一个token）会导致序列太长、训练效率低，且无法处理多字词的语义。按词切分又面临词表太大、OOV（未知词）问题。

**Q：BPE算法的核心思想是什么？**
从字符级别开始，反复合并出现频率最高的相邻字符对，逐步构建词表。最终词表大小可控（几万到十几万），且能处理任何文本（最差退化到字符级）。

**Q：不同模型的分词器能混用吗？**
不能。每个模型有自己的词表和分词规则，GPT-4、Claude、Qwen的Tokenizer互不兼容。同一句话在不同模型可能切成不同数量的token。

### 类比记忆

> 🎯 **类比：切蛋糕**
>
> - **按字切** = 每次切一小块 — 太碎了，吃100口才吃完
> - **按词切** = 每次切一大块 — 快但有些蛋糕形状不规则切不了
> - **BPE** = 先切小块，发现经常连在一起的就合并成大块 — 平衡了碎片化和灵活性
>
> "unhappiness" → ["un", "happiness"] 而不是 ["u","n","h","a","p","p","i","n","e","s","s"]

### 深度拆解

**主流分词算法：**

| 算法 | 原理 | 代表模型 |
|:---:|------|:---:|
| BPE | 从字符开始，反复合并高频相邻对 | GPT系列、LLaMA |
| WordPiece | 类似BPE，但用似然值而非频率决定合并 | BERT |
| Unigram | 从大词表开始，逐步删除低频子词 | T5、ALBERT |
| SentencePiece | BPE/Unigram + 原生多语言支持 | Qwen、LLaMA |

**BPE构建过程示例：**

```
初始: {"low":5, "lower":2, "newest":6, "widest":3}

Step 1: 拆成字符
  l o w </w>, l o w e r </w>, n e w e s t </w>, ...

Step 2: 统计相邻字符对频率，合并最高频
  "e"+"s" 出现9次 → 合并 "es"

Step 3: 继续合并
  "es"+"t" → "est"
  "est"+"</w>" → "est</w>"

... 反复直到词表达到目标大小
```

**中文的特殊性：**
- 英文天然有空格分隔，BPE在词内合并子词
- 中文没有空格，SentencePiece直接在整句层面做分词
- Qwen等中文模型通常一个汉字≈1-2个token，英文一个单词≈1-2个token

### 面试话术

> "分词器把文本切分成Token，是模型处理文本的第一步。主流算法是BPE——从字符级开始，反复合并出现频率最高的相邻字符对，构建出大小可控的词表。优点是能处理任何文本（最差退化到字符级），且词表大小合理。
>
> 不同模型的Tokenizer不兼容，GPT-4、Claude、Qwen各有自己的词表和分词规则。中文场景下用SentencePiece较多，因为它原生支持无空格文本。分词质量直接影响模型的序列长度和效率——同样一句话，切出的token越少越省钱。"

## Q6：大模型是怎么训练出来的？

### 一句话破题

大模型训练分四步走：预训练（学语言和知识）→ SFT（学遵循指令）→ RLHF/DPO（对齐人类偏好）→ 模型评估迭代。每一步的目标和数据完全不同。

### 类比记忆

> 🎯 **类比：培养一名大学生**
>
> - **预训练** = 读了从小学到大学所有课本 — 积累基础知识和语言能力
> - **SFT** = 实习时学工作规范 — 学会按领导要求做事
> - **RLHF/DPO** = 360度绩效评估 — 知道什么是好的，什么是不该做的
> - **评估迭代** = 毕业论文答辩 — 检验能力，发现问题继续改进

### 深度拆解

**四阶段训练Pipeline：**

```
阶段1: 预训练 (Pre-training)
├─ 数据: 万亿级token的海量互联网文本
├─ 目标: Next Token Prediction (预测下一个词)
├─ 硬件: 千卡A100/H100集群，训练数月
├─ 产出: 基座模型 (Base Model)
└─ 特点: 会续写文本，但不会对话

阶段2: 指令微调 (SFT)
├─ 数据: 数万~百万条「指令-回答」对
├─ 目标: 学会遵循人类指令格式回答
├─ 硬件: 数卡~数十卡，训练数小时~数天
├─ 产出: 指令模型 (Chat Model)
└─ 特点: 会对话了，但回答质量参差不齐

阶段3: 偏好对齐 (RLHF/DPO)
├─ 数据: 人类标注的「好回答 vs 坏回答」偏好对
├─ 目标: 让模型输出更安全、有用、符合人类偏好
├─ 方法: PPO (强化学习) 或 DPO (直接偏好优化)
├─ 产出: 对齐模型 (Aligned Model)
└─ 特点: 回答质量稳定，安全性高

阶段4: 评估与迭代
├─ 评估: MMLU、HumanEval、MT-Bench等基准
├─ 分析: 发现薄弱环节，补充训练数据
└─ 迭代: 回到对应阶段继续优化
```

**预训练的关键细节：**

| 要素 | 说明 |
|:---:|------|
| 数据 | 网页爬虫、书籍、论文、代码、对话等，去重+质量过滤 |
| 目标函数 | 因果语言建模（预测下一个token的交叉熵损失） |
| 优化器 | AdamW，学习率warmup+cosine衰减 |
| 并行策略 | 数据并行 + 张量并行 + 流水线并行（3D并行） |
| 批次大小 | 百万级token的effective batch size |

### 面试话术

> "大模型训练分四个阶段。预训练用万亿级token的海量文本做next token prediction，产出基座模型——会续写但不会对话。SFT用数万到百万条指令-回答对微调，让模型学会遵循指令格式。RLHF或DPO用人类偏好数据做对齐，让回答更安全有用。最后是评估迭代，用MMLU等基准测试发现薄弱环节。
>
> 预训练是最耗资源的阶段，需要千卡集群训练数月。SFT和对齐相对轻量，但也需要精心构造的高质量数据。整个Pipeline的数据质量比数据数量更重要。"

## Q7：什么是Scaling Law？大模型的「涌现能力」是怎么回事？

### 一句话破题

Scaling Law是指模型能力随参数量、数据量、计算量呈可预测的幂律增长；涌现能力是指模型规模达到某个临界点后，突然出现小模型不具备的能力——前者是"量变引起质变"的规律，后者是质变的具体表现。

### 核心三问

**Q：Scaling Law的具体公式是什么？**
Loss ≈ A/N^α + B/D^β + C，其中N是参数量，D是数据量，α和β是幂指数。核心结论：要最优地分配计算预算，参数量和数据量要按比例同时增长。

**Q：涌现能力有哪些典型例子？**
少样本学习（few-shot）、思维链推理（CoT）、代码理解、指令遵循——这些能力在7B以下模型几乎为零，在70B+模型上突然出现。

**Q：涌现能力是真的"突变"还是测量假象？**
存在争议。有研究认为部分涌现是评估指标的离散性导致的假象（如精确匹配），如果用连续指标可能就是平滑增长。但少样本学习和CoT等能力的确实质提升明显。

### 类比记忆

> 🎯 **类比：烧开水**
>
> - **Scaling Law** = 温度随火力增大而升高 — 可预测的线性/幂律关系
> - **涌现能力** = 100°C时水突然沸腾 — 到了临界点质变
>
> 你不能说"90°C的水是90%的沸水"，因为沸腾是一个质变过程。同理，某些AI能力不是随规模平滑增长的，而是在某个临界点"突然会了"。

### 深度拆解

**Scaling Law的核心结论：**

```
计算预算 C = 6 × N × D
  N = 参数量
  D = 训练token数

最优分配: N 和 D 按比例增长
  → 算力翻倍 → 参数量增√2倍，数据量增√2倍

Chinchilla法则: 最优token数 ≈ 20 × 参数量
  → 70B模型应该用1.4万亿token训练
```

**典型的涌现能力：**

| 能力 | 临界规模 | 表现 |
|:---:|:---:|------|
| Few-shot学习 | ~10B | 看几个例子就能学会新任务 |
| 思维链推理 | ~60B | "Let's think step by step"突然有效 |
| 指令遵循 | ~7B | 能理解复杂多步指令 |
| 代码生成 | ~30B | 能写出可运行的复杂代码 |
| 多语言翻译 | ~10B | 零样本跨语言迁移 |

### 面试话术

> "Scaling Law是OpenAI提出的经验规律：模型Loss随参数量N和数据量D呈幂律下降，公式是L≈A/N^α+B/D^β+C。核心实践指导是Chinchilla法则——最优训练token数约为参数量的20倍，参数和数据要按比例增长。
>
> 涌现能力是指模型规模超过临界点后突然出现的能力，如few-shot学习和CoT推理在60B+模型才明显。关于涌现是否是'突变'存在争议——有研究认为部分涌现是离散评估指标造成的假象，但少样本学习等能力确实是质的飞跃。
>
> 这两个概念的关系是：Scaling Law描述量变规律，涌现能力是量变到质变的具体表现。"

## Q8：大模型微调的方案有哪些？

### 一句话破题

微调方案按「改多少参数」分三档：全量微调（改全部参数，效果最好但贵）、参数高效微调PEFT（改0.1-1%参数，性价比最高）、提示微调（几乎不改参数，最轻量）。

### 类比记忆

> 🎯 **类比：装修房子**
>
> - **全量微调** = 推倒重建 — 效果最好，但成本最高，工期最长
> - **LoRA** = 软装改造 — 换窗帘、贴壁纸、摆家具，不动承重墙，省钱又有效
> - **Prefix Tuning** = 改灯光氛围 — 只调灯光不改结构，最轻量但效果有限
> - **冻结+少量调** = 只换门把手 — 改动极小，效果也极有限

### 深度拆解

**微调方案全景对比：**

| 方案 | 可训练参数 | 显存需求 | 效果 | 适用场景 |
|:---:|:---:|:---:|:---:|------|
| Full Fine-tuning | 100% | 极高（80GB+） | 最好 | 资源充足、追求极致效果 |
| LoRA | 0.1-1% | 低（单卡可行） | 接近全量 | 主流选择，性价比最高 |
| QLoRA | 0.1%+量化 | 极低（消费级GPU） | 略低于LoRA | 资源受限场景 |
| Prefix Tuning | <0.1% | 极低 | 中等 | 轻量适配 |
| P-Tuning v2 | <0.1% | 低 | 中等 | 中文场景较多 |
| Adapter | ~1% | 低 | 中等 | 早期PEFT方案 |
| Freeze(冻结) | 极少 | 低 | 有限 | 快速实验 |

**LoRA为什么是主流？**

| 优势 | 说明 |
|:---:|------|
| 推理零开销 | 训练后A/B矩阵可合并回W，推理时和原模型一样 |
| 模块化 | 一个基座模型+多套LoRA，热切换不同能力 |
| 不遗忘 | 原始权重冻结，通用能力保留 |
| 训练稳定 | 可训练参数少，对学习率不敏感 |
| 可组合 | 多个LoRA可加权混合，无需重训 |

### 面试话术

> "微调方案按改多少参数分三档。全量微调改全部参数，效果最好但需要80GB+显存，一般只有大厂用得起。PEFT方法只改0.1-1%的参数，其中LoRA是绝对主流——旁路低秩分解，训练后可合并回原始权重实现零推理开销，还能一个基座配多套LoRA热切换。QLoRA在LoRA基础上加4bit量化，消费级GPU也能跑。
>
> 选型建议：资源充足追求极致用全量微调，绝大多数场景用LoRA，资源极度受限用QLoRA。Prefix Tuning和Adapter是早期方案，现在基本被LoRA取代。"

## Q9：请讲一下LoRA技术，除了减少参数量，它还有哪些优点？

### 一句话破题

LoRA（Low-Rank Adaptation）在冻结的原始权重旁加两个小矩阵A和B做低秩分解——除了省参数，还有推理零开销、模块化部署、低遗忘风险、训练稳定、权重可混合五大优点。

### 核心三问

**Q：LoRA的数学原理是什么？**
原始权重W（d×d）冻结不动，旁路加两个小矩阵A（d×r）和B（r×d），输出 = x·(W + α·(B·A))。r远小于d（通常8-16），参数量从d²降到2dr，减少100倍以上。

**Q：为什么叫"低秩"？**
研究发现微调时权重的变化量ΔW具有内在低秩性——虽然ΔW是d×d的大矩阵，但其有效信息维度（秩）通常只有8-16，用两个小矩阵就能很好近似。

**Q：推理时真的零开销吗？**
是的。训练完成后把B·A直接加到W上（W' = W + α·BA），之后推理用的就是W'，和原始模型计算量完全一致。不像Adapter要在每层加额外网络导致延迟叠加。

### 类比记忆

> 🎯 **类比：在书上做笔记**
>
> - **全量微调** = 把书重新排版印刷 — 贵且耗时
> - **LoRA** = 在书页空白处贴便利贴 — 原文不动，贴修正和补充
> - **合并** = 把便利贴内容抄进正文，之后看的就是一本完整的书
>
> 便利贴很小（低秩），但写满了关键修正。看完后抄进正文（合并），读起来和原版一样快（零开销）。

### 深度拆解

**LoRA核心公式：**

```
原始: output = x @ W           (W: d×d, 冻结)
LoRA: output = x @ (W + α·(B@A))  (A: d×r, B: r×d, 可训练)

参数量对比 (d=4096, r=16):
  原始更新: 4096×4096 = 16,777,216
  LoRA:     4096×16 + 16×4096 = 131,072
  减少倍数: ~128倍
```

**五大优点详解：**

| 优点 | 原理 | 工程价值 |
|:---:|------|------|
| 推理零开销 | B·A合并入W，推理计算图和原模型一致 | 不像Adapter有额外延迟 |
| 模块化部署 | 基座模型14GB常驻 + 每套LoRA几十MB | 一台机器服务多个场景 |
| 低遗忘风险 | W全程冻结，知识在旁路学习 | 通用能力保留率高于全量微调 |
| 训练稳定 | 梯度搜索空间小，对学习率不敏感 | 调参成本低，不容易跑飞 |
| 权重可混合 | W' = W + α₁·B₁A₁ + α₂·B₂A₂ | 多个LoRA加权混合，无需重训 |

**LoRA vs 全量微调 vs Adapter：**

| 维度 | 全量微调 | Adapter | LoRA |
|:---:|:---:|:---:|:---:|
| 可训练参数 | 100% | ~1% | ~0.1-1% |
| 推理开销 | 无 | 有 | **无** |
| 遗忘风险 | 高 | 低 | **低** |
| 部署灵活性 | 低 | 中 | **高** |
| 训练稳定性 | 差 | 好 | **好** |
| 权重可组合 | ❌ | ❌ | **✅** |
| 效果上限 | 最高 | 中 | 接近全量 |

### 面试话术

> "LoRA在冻结的原始权重旁加两个低秩小矩阵A和B，输出=x·(W+α·BA)。参数量减少100倍以上，但更重要的是五个工程优点：
>
> 第一，推理零开销——训练后B·A合并入W，推理和原模型完全一致。第二，模块化部署——一个14GB基座配多套几十MB的LoRA，按需热切换。第三，低遗忘——W冻结不更新，通用能力保留好。第四，训练稳定——可训练参数少，对超参不敏感。第五，权重可混合——多个LoRA加权组合，无需重训就能融合多种能力。
>
> LoRA能成为PEFT事实标准，不是单一优势，而是这五个优点叠加的结果，把Adapter等早期方案完全比下去了。"

## Q10：SFT之后还有哪些Post-Training？RLHF、DPO、GRPO、拒绝采样什么关系？

### 一句话破题

Post-Training是SFT之后的所有优化阶段统称，目标是让模型输出更符合人类偏好——RLHF用强化学习做、DPO用直接偏好优化做、GRPO是RLHF的无价值模型版本、拒绝采样是数据筛选策略。

### 类比记忆

> 🎯 **类比：培训新员工**
>
> - **SFT** = 新人培训 — 学会基本工作流程和规范
> - **RLHF** = 导师带教+绩效考核 — 导师打分（奖励模型），根据分数调整行为
> - **DPO** = 对比学习 — 直接给看「好回答 vs 坏回答」，自己悟差距
> - **GRPO** = 小组互评 — 不需要导师，组内互相打分排名
> - **拒绝采样** = 精选作业 — 写10份作业选最好的2份当范例

### 深度拆解

**Post-Training方法关系图：**

```
                Post-Training
                /     |       \
          RLHF系列   DPO系列   其他
          /    \       |        |
        PPO   GRPO    DPO    拒绝采样
        |      |       |
     价值模型  无价值   无RL
     +奖励模型 模型     直接优化
```

**四种方法对比：**

| 方法 | 核心思路 | 需要奖励模型 | 训练稳定性 | 代表应用 |
|:---:|------|:---:|:---:|:---:|
| RLHF(PPO) | 训练奖励模型→用PPO优化策略 | ✅ | 差（PPO难调） | GPT-3.5/4, InstructGPT |
| DPO | 直接用偏好对优化，跳过奖励模型 | ❌ | 好 | LLaMA-3, Zephyr |
| GRPO | 组内采样+相对排名做奖励 | ❌ | 中等 | DeepSeek-R1 |
| 拒绝采样 | 生成多回答→选最好的→加入SFT数据 | 需要评分 | 好 | LLaMA-2, DeepSeek |

**各方法详解：**

**RLHF（PPO）：**
```
① 训练奖励模型: 用人类偏好数据训练一个打分模型
② PPO优化:
   - 模型生成回答
   - 奖励模型打分
   - 用PPO算法优化模型策略（最大化奖励）
   - KL散度约束防止偏离SFT模型太远
```

**DPO：**
```
核心洞察: 偏好优化可以转化为分类问题
① 不训练奖励模型
② 直接用偏好对(y_good, y_bad)优化
③ 损失函数: 让模型给y_good的概率高于y_bad
④ 公式: L = -log σ(β·(log π(y_good)/π_ref - log π(y_bad)/π_ref))
```

**GRPO（DeepSeek-R1使用）：**
```
① 对同一问题生成G个回答
② 用规则/验证器给每个回答打分
③ 组内归一化做相对排名
④ 用排名作为奖励信号优化模型
⑤ 不需要单独训练奖励模型
```

**拒绝采样（Rejection Sampling）：**
```
① 对每个问题生成N个回答
② 用奖励模型/规则打分
③ 保留top-K个高质量回答
④ 将这些高质量对加入SFT数据
⑤ 重新做SFT
```

### 面试话术

> "Post-Training是SFT之后让模型对齐人类偏好的阶段。RLHF是最经典的方法——先训练奖励模型，再用PPO做强化学习优化，效果最好但训练不稳定。DPO跳过奖励模型，直接用偏好对做分类式优化，简单稳定，LLaMA-3等模型在用。GRPO是DeepSeek-R1的方法，组内采样+相对排名做奖励，不需要单独训奖励模型。拒绝采样是数据策略——生成多个回答选最好的加入训练数据，LLaMA-2用了这个方法。
>
> 趋势上DPO因为简单稳定越来越流行，RLHF在追求极致效果时仍有一席之地。实际项目中经常组合使用——先拒绝采样提升数据质量，再DPO做偏好对齐。"

## Q11：大模型的DPO和PPO的区别是什么？

### 一句话破题

PPO需要训练奖励模型+用强化学习优化，链路长且不稳定；DPO直接用偏好对做分类优化，跳过奖励模型，简单稳定——两者目标相同（对齐人类偏好），但实现路径完全不同。

### 核心三问

**Q：DPO为什么不需要奖励模型？**
数学上证明了：在特定的偏好模型假设下，最优策略可以直接从偏好数据中推导出来，不需要显式训练奖励模型。DPO把这个推导变成了一个简单的分类损失。

**Q：PPO效果比DPO好吗？**
不一定。PPO的理论上限更高（奖励模型可以捕捉复杂偏好），但调参困难。DPO虽然简单但效果稳定，很多实践表明DPO效果接近甚至超过PPO。

**Q：什么场景该选哪个？**
资源充足+追求极致效果→PPO；快速迭代+稳定性优先→DPO；有明确验证规则（如数学/代码）→GRPO。

### 类比记忆

> 🎯 **类比：教小孩学做饭**
>
> - **PPO（RLHF）** = 先培训一个美食评委（奖励模型），让评委品尝打分，小孩根据分数调整做法 — 评委水平决定效果上限，但整个链路复杂
> - **DPO** = 直接给小孩看两盘菜："这盘好吃，那盘不好吃"，让他自己悟差距 — 简单直接，但没有专业评委的细致反馈

### 深度拆解

**PPO vs DPO 全方位对比：**

| 维度 | PPO (RLHF) | DPO |
|:---:|:---:|:---:|
| 需要奖励模型 | ✅ 需要单独训练 | ❌ 不需要 |
| 训练阶段 | 4阶段（SFT→RM→PPO→迭代） | 2阶段（SFT→DPO） |
| 训练稳定性 | 差（PPO超参敏感） | 好（类似SFT） |
| 计算开销 | 高（在线生成+4个模型） | 低（离线数据+2个模型） |
| 效果上限 | 理论更高 | 接近PPO |
| 实现复杂度 | 高 | 低 |
| 代表应用 | GPT-4, InstructGPT | LLaMA-3, Zephyr |

**PPO的四个模型：**
```
① Actor Model — 当前策略，生成回答
② Reference Model — 冻结的SFT模型，算KL散度
③ Reward Model — 给回答打分
④ Critic Model — 估计状态价值
→ 需要同时加载4个模型，显存巨大
```

**DPO的两个模型：**
```
① Policy Model — 当前策略（可训练）
② Reference Model — 冻结的SFT模型
→ 只需2个模型，显存需求减半
```

### 面试话术

> "PPO和DPO的目标相同——让模型输出符合人类偏好，但路径不同。PPO需要先训练奖励模型，再用PPO算法做强化学习优化，同时维护Actor、Reference、Reward、Critic四个模型，链路长、调参难、显存大。DPO从数学上证明了可以跳过奖励模型，直接用偏好对做分类优化，只需2个模型，训练过程类似SFT，简单稳定。
>
> 实践中DPO越来越流行，LLaMA-3、Zephyr等模型都用了DPO。PPO在追求极致效果时仍有价值，特别是需要在线学习场景。选型上：快速迭代用DPO，资源充足追求极致用PPO，有验证规则的场景（数学/代码）用GRPO。"

## Q12：大模型生成文本时的解码策略有哪些？

### 一句话破题

解码策略决定模型每一步「从概率分布中怎么选下一个词」——贪心选最大概率（确定性但单调）、Beam Search维护top-K路径（质量高但慢）、采样引入随机性（有多样性但可能跑偏）。

### 类比记忆

> 🎯 **类比：走迷宫**
>
> - **贪心** = 每步都走最宽的路 — 快但可能错过最优路线
> - **Beam Search** = 同时走K条最宽的路，走到底选最好的 — 质量高但费时
> - **采样** = 按路宽的概率随机选 — 有惊喜但也可能走死胡同
> - **Top-K采样** = 只在K条最宽的路里随机选 — 平衡质量和多样性
> - **Top-P采样** = 只在累计概率P内的路里随机选 — 更灵活的Top-K

### 深度拆解

**五种解码策略对比：**

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|:---:|------|:---:|:---:|------|
| 贪心 | 每步选概率最大的词 | 快、确定 | 重复、单调 | 翻译、摘要 |
| Beam Search | 维护top-B个候选序列 | 质量高 | 慢、缺乏多样性 | 机器翻译 |
| 纯采样 | 按概率分布随机采样 | 多样性最强 | 可能不连贯 | 创意写作 |
| Top-K采样 | 只在top-K个词中采样 | 平衡性较好 | K固定不灵活 | 通用 |
| Top-P采样 | 只在累计概率P内采样 | 自适应 | 需调P值 | 对话（主流） |

**Top-P（核采样）为什么最主流：**

```
场景1: "中国的首都是" → 下一个词概率: 北京(0.99) 上海(0.005)...
  Top-P=0.9 → 只选"北京"（因为北京一个就超过0.9）
  → 确定性输出，正确

场景2: "今天天气真好，我想去" → 下一个词概率: 公园(0.2) 散步(0.15) 看电影(0.12)...
  Top-P=0.9 → 选多个词直到累计概率0.9
  → 有多样性的合理选择
```

> 💡 Top-P的自适应性：高概率词少选几个，低概率词多选几个，比固定K更合理。

### 面试话术

> "解码策略决定模型怎么从概率分布中选词。贪心每步选最大概率，快但容易重复。Beam Search维护top-B路径选最优，质量高但慢且缺乏多样性。采样类策略引入随机性，其中Top-P（核采样）最主流——只在累计概率P的词中采样，高概率时近乎确定性，低概率时有多样性，自适应性强。
>
> 实际应用中，翻译/摘要用贪心或Beam Search（追求准确），对话/创作用Top-P采样（追求自然和多样性）。"

## Q13：大模型的温度值、Top-P、Top-K分别是什么？各场景最佳设置？

### 一句话破题

温度值控制概率分布的「尖锐度」（低温确定，高温随机），Top-K限制候选词数量，Top-P限制候选词累计概率——三者都是控制生成随机性的旋钮，通常组合使用。

### 类比记忆

> 🎯 **类比：选餐厅**
>
> - **温度(T)** = 你有多愿意尝试新餐厅
>   - T=0 → 永远去最熟悉的那家（确定性）
>   - T=1 → 按知名度比例随机选（自然）
>   - T=2 → 去很奇怪的餐厅也行（混乱）
>
> - **Top-K** = 只在排名前K的餐厅里选
> - **Top-P** = 只在累计口碑P%的餐厅里选

### 深度拆解

**三个参数详解：**

**温度值（Temperature）：**
```
原始概率: [0.7, 0.2, 0.1]
T=0.1 → [0.998, 0.002, 0.000]  (极度尖锐，近乎贪心)
T=1.0 → [0.7, 0.2, 0.1]        (不变)
T=2.0 → [0.42, 0.31, 0.27]     (扁平化，更随机)

公式: p_i' = exp(log(p_i)/T) / Σexp(log(p_j)/T)
```

**Top-K：**
```
词表概率: [0.4, 0.3, 0.15, 0.1, 0.03, 0.02, ...]
K=3 → 只在 [0.4, 0.3, 0.15] 中采样，其余丢弃
```

**Top-P（核采样）：**
```
词表概率: [0.4, 0.3, 0.15, 0.1, 0.03, 0.02, ...]
P=0.8 → 0.4+0.3=0.7 < 0.8, 再加0.15=0.85 > 0.8
       → 在 [0.4, 0.3, 0.15] 中采样
```

**各场景推荐设置：**

| 场景 | Temperature | Top-P | Top-K | 说明 |
|:---:|:---:|:---:|:---:|------|
| 代码生成 | 0-0.2 | 0.9 | 0(不限) | 追求准确，低温高确定性 |
| 数学推理 | 0-0.1 | 0.9 | 0 | 逻辑严谨，几乎贪心 |
| 翻译/摘要 | 0.1-0.3 | 0.9 | 0 | 准确为主，少量灵活 |
| 问答/对话 | 0.5-0.7 | 0.9 | 0 | 自然且有轻微多样性 |
| 创意写作 | 0.7-1.0 | 0.95 | 0 | 多样性和创造力 |
| 头脑风暴 | 1.0-1.2 | 0.97 | 0 | 最大化多样性 |

> ⚠️ **注意：** Top-P和Top-K通常设一个即可，不需要同时限制。温度和Top-P配合使用最常见。API调用时通常默认Temperature=1, Top-P=1。

### 面试话术

> "温度值控制概率分布的尖锐度——低温让高概率词更突出（接近贪心），高温让分布更平均（更随机）。Top-K限制候选词数量为固定的K个，Top-P限制候选词的累计概率为P，后者更自适应。
>
> 场景设置上：代码和数学用低温（0-0.2）追求准确，对话用中温（0.5-0.7）追求自然，创意写作用高温（0.7-1.0）追求多样性。Top-P通常设0.9-0.95，Top-K设0（不限制）让Top-P自己适配。
>
> 实际API调用中，温度是最常用的旋钮，Top-P作为辅助。不建议同时设Top-K和Top-P，用一个就够。"

## Q14：KV Cache是什么？Prompt Caching的原理是什么？

### 一句话破题

KV Cache是**单次推理内**的优化——缓存已生成token的K/V矩阵避免重复计算，把复杂度从O(N³)降到O(N²)；Prompt Caching是**跨请求**的优化——不同请求共享相同前缀的KV Cache，跳过重复计算省钱降延迟。

### 核心三问

**Q：为什么自回归生成需要KV Cache？**
自回归生成每生成一个新token都要对所有已有token做attention。如果不缓存，第i步要重新计算前i-1个token的K/V，总计算量O(N³)。缓存后每步只算新token，总计算量O(N²)。

**Q：KV Cache的显存代价有多大？**
7B模型（32层×32头×128维），序列长度32K时KV Cache约17GB，比模型权重（14GB）还大。这是LLM工程的核心瓶颈，催生了PagedAttention、KV量化、MQA/GQA等优化。

**Q：Prompt Caching怎么省钱的？**
Claude：缓存命中只收10%的token费用，写入收1.25倍。OpenAI：自动缓存，命中减半。固定System Prompt场景下，第二次请求起省钱90%。

### 类比记忆

> 🎯 **类比：考试做题**
>
> - **没有KV Cache** = 每做一道新题，把前面所有题重新算一遍 — 越往后越慢
> - **KV Cache** = 前面的计算过程记在草稿纸上，新题只算新内容 — 速度稳定
> - **Prompt Caching** = 你的学霸同桌把公共题目的解题过程借给你抄 — 省大量时间
>
> 草稿纸越大记得越多（KV Cache显存大），但草稿纸也占桌面空间（显存压力）。

### 深度拆解

**KV Cache原理：**

```
朴素自回归（无KV Cache）:
  Step 1: 算[P]的attention → output t1     计算量: O(1²)
  Step 2: 算[P,t1]的attention → output t2  计算量: O(2²)
  ...
  Step N: 算[P,t1...tN-1]的attention       计算量: O(N²)
  总计: O(N³) ← 不可接受

带KV Cache:
  Step 1: 算[P]的K,V → 存入cache → 算attention → t1
  Step 2: 只算[t1]的K,V → 追加到cache → 用cache算attention → t2
  ...
  Step N: 只算[tN-1]的K,V → 追加到cache → 用cache算attention → tN
  总计: O(N²) ← 可接受
```

**KV Cache显存公式：**
```
显存 = 2(K+V) × B(batch) × N(序列长度) × L(层数) × H(头数) × d(每头维度) × 2字节(FP16)

7B模型示例: 2×1×32000×32×32×128×2 ≈ 17GB
```

**Prompt Caching工作机制：**

```
请求1: System Prompt(3000 tokens) + "试用期最长多久？"
  → 计算System Prompt的KV Cache → 正常计费
  → 额外保留KV Cache在服务器

请求2: System Prompt(3000 tokens) + "必备条款有哪些？"
  → 发现System Prompt前缀完全匹配 → 直接复用KV Cache
  → 只算新增部分 → 省下3000 tokens的计算
  → 费用: 命中部分只收10%
```

**工程陷阱：**

| 陷阱 | 说明 | 正确做法 |
|:---:|------|------|
| 动态内容在前 | 日期/用户名放前面，整个前缀每次都变 | 固定内容放前面，动态内容放后面 |
| 前缀不完全一致 | 多一个空格就miss | 严格保持前缀一致性 |
| 低流量失效 | Claude缓存5分钟过期 | 高流量场景才划算 |

### 面试话术

> "KV Cache和Prompt Caching是同一套缓存思路在两个时间尺度上的应用。
>
> KV Cache是单次推理内的优化——自回归生成每步需要所有已有token的K/V做attention，缓存后每步只算新token的K/V，总计算量从O(N³)降到O(N²)，是让自回归生成可行的基本盘。但KV Cache显存很大，7B模型32K序列就17GB，催生了PagedAttention和KV量化等优化。
>
> Prompt Caching是跨请求的优化——不同请求共享相同前缀的KV Cache。Claude用显式cache_control断点标记，命中收10%费用；OpenAI自动缓存。适合'固定System Prompt + 动态问题'的场景，如客服AI。工程上要注意固定内容放前面、动态内容放后面，前缀差一个字符就miss。"

## Q15：大模型量化是什么？INT8/INT4/AWQ/GPTQ怎么选？

### 一句话破题

量化是把模型权重从高精度（FP16）压缩到低精度（INT8/INT4），用更少bit表示相同信息——直接效果是显存减半到1/4，推理速度提升，代价是精度轻微下降。

### 核心三问

**Q：量化为什么能省显存？**
FP16每个参数占2字节，INT8占1字节，INT4占0.5字节。7B模型FP16需14GB，INT8需7GB，INT4只需3.5GB——消费级GPU也能跑了。

**Q：量化会掉点吗？**
会，但程度不同。INT8量化通常掉点很小（<1%），INT4量化掉点更明显。好的量化方法（AWQ、GPTQ）能把INT4的掉点控制在可接受范围。

**Q：AWQ和GPTQ有什么区别？**
GPTQ基于二阶Hessian信息做逐层量化，精度好但量化速度慢。AWQ通过识别「重要权重」保护关键参数，量化快且精度好。实际选AWQ更多。

### 类比记忆

> 🎯 **类比：图片压缩**
>
> - **FP16** = RAW无损照片 — 质量最好但文件大
> - **INT8** = JPEG高质量 — 几乎看不出差别，文件减半
> - **INT4** = JPEG中质量 — 细微处有损失，但文件只有1/4
> - **AWQ** = 智能压缩 — 人脸区域保持高质量，背景压缩更多（保护重要权重）
> - **GPTQ** = 精密计算压缩 — 逐像素优化，质量最好但压缩速度慢

### 深度拆解

**量化方案对比：**

| 方案 | 位宽 | 显存(7B) | 精度损失 | 量化速度 | 推理速度 |
|:---:|:---:|:---:|:---:|:---:|:---:|
| FP16（基线） | 16bit | 14GB | 0 | - | 1× |
| INT8（W8A8） | 8bit | 7GB | <1% | 快 | 1.5-2× |
| INT4（W4A16） | 4bit | 3.5GB | 1-3% | 中 | 2-3× |
| AWQ-INT4 | 4bit | 3.5GB | 1-2% | 快 | 2-3× |
| GPTQ-INT4 | 4bit | 3.5GB | 0.5-2% | 慢 | 2-3× |

**量化分类：**

| 类型 | 说明 | 适用场景 |
|:---:|------|------|
| 权重量化(W-only) | 只量化权重，激活保持FP16 | 推理为主 |
| 权重+激活量化(W+A) | 权重和激活都量化 | 追求极致速度 |
| PTQ（训练后量化） | 模型训练完直接量化 | 最常用，无需重新训练 |
| QAT（量化感知训练） | 训练时就模拟量化 | 精度要求极高 |

**选型决策：**

```
显存够用(>20GB)？
  └─ YES → FP16或INT8，精度优先
  └─ NO
      ├─ 消费级GPU(8-16GB)？
      │    └─ AWQ-INT4（量化快、精度好、生态好）
      ├─ 追求极致精度？
      │    └─ GPTQ-INT4（精度最优但量化慢）
      └─ 需要部署到手机/边缘设备？
           └─ INT4 + 进一步蒸馏
```

### 面试话术

> "量化是把权重从FP16压缩到INT8或INT4，直接效果是显存减半到1/4。7B模型从14GB降到3.5GB（INT4），消费级GPU就能跑。
>
> 选型上：INT8量化掉点很小（<1%），适合显存稍紧张的场景。INT4量化更激进但AWQ和GPTQ能把掉点控制在1-3%可接受范围。AWQ通过保护重要权重实现快速量化+好精度，是目前最常用的INT4方案。GPTQ基于Hessian信息逐层优化，精度最好但量化慢。
>
> 实际项目推荐AWQ-INT4——量化快、精度好、vLLM等推理框架支持好。如果显存够用就不量化或用INT8，精度优先。"

## Q16：如何写好Prompt？分享下Prompt工程实践经验？

### 一句话破题

好Prompt的核心是「清晰表达意图 + 给足上下文 + 定义输出格式」——不是让模型猜你要什么，而是把需求讲清楚到任何人都能理解的程度。

### 类比记忆

> 🎯 **类比：给实习生布置任务**
>
> 差的Prompt = "帮我做个分析" — 实习生完全不知道做什么
> 好的Prompt = "分析这份销售数据，按区域统计月度增长率，用表格输出，重点关注下降超过10%的区域" — 清晰、具体、有格式要求

### 深度拆解

**Prompt工程六大原则：**

| 原则 | 说明 | 示例 |
|:---:|------|------|
| 角色定义 | 给模型一个专家身份 | "你是一位资深Python工程师" |
| 任务清晰 | 具体说明要做什么 | "审查以下代码的安全漏洞" |
| 上下文充分 | 给出背景信息 | "这是一个金融交易系统的支付模块" |
| 输出格式 | 指定输出结构 | "用JSON格式输出，包含severity和description字段" |
| 示例引导 | 给Few-shot示例 | "示例：输入X → 输出Y" |
| 约束条件 | 说明限制和边界 | "不超过200字，不要使用专业术语" |

**Prompt模板示例：**

```

你是一位资深代码审查专家，擅长发现安全漏洞和性能问题。

# 任务
审查以下Python代码，找出潜在问题。

这是一个电商平台的支付回调处理模块，每天处理10万笔交易。

```python
{code}
```

# 输出格式
请按以下JSON格式输出：
{
  "issues": [
    {
      "type": "安全|性能|规范|逻辑",
      "severity": "高|中|低",
      "line": 行号,
      "description": 问题描述,
      "suggestion": 修改建议
    }

- 只报告真实问题，不要过度解读
- 每个问题必须给出可执行的修改建议
```

**高级Prompt技巧：**

| 技巧 | 适用场景 | 说明 |
|:---:|:---:|------|
| CoT（思维链） | 推理/数学 | "请一步步思考" |
| Few-shot | 格式控制 | 给2-3个输入输出示例 |
| 自我反思 | 质量提升 | "请检查你的回答是否有误" |
| 角色扮演 | 专业领域 | "你是XX领域的专家" |
| 分步指令 | 复杂任务 | 拆成多个步骤依次执行 |

### 面试话术

> "写好Prompt的核心是清晰表达意图、给足上下文、定义输出格式。我的实践经验有六个原则：定义角色（你是XX专家）、任务清晰（具体做什么）、上下文充分（背景信息）、输出格式（JSON/表格/指定结构）、示例引导（Few-shot）、约束条件（限制和边界）。
>
> 复杂任务我会用CoT让模型一步步推理，格式控制用Few-shot示例，质量要求高的加自我反思步骤。最关键的是把Prompt写到'任何人类看了都知道要做什么'的程度——如果你自己看了都要猜，模型肯定也会猜错。"

## Q17：什么是CoT？为什么效果好？有什么局限性？

### 一句话破题

CoT（Chain of Thought）让模型在给出答案前先写出推理过程——效果好的本质是「给模型更多token来计算」，把隐式推理变成显式文字，降低每步推理难度。

### 核心三问

**Q：CoT为什么有效？**
两个假说：①模型生成中间推理token相当于增加了计算量（更多forward pass）；②显式写出步骤让模型不会跳步，每步的预测任务更简单。

**Q：什么场景CoT最有效？**
多步推理任务（数学题、逻辑推理、代码调试），模型规模需要足够大（通常60B+才有效，小模型CoT反而更差）。

**Q：CoT的局限性是什么？**
不保证推理链正确（中间步骤可能出错且错误会传播）、增加token消耗和延迟、小模型无效、简单任务不需要。

### 类比记忆

> 🎯 **类比：心算 vs 草稿纸**
>
> - **直接回答** = 心算 "37×24=?" — 容易算错
> - **CoT** = 在草稿纸上列竖式 — 每步只算简单的一位数乘法，不容易错
>
> 草稿纸（CoT）不会让你变聪明，但让每一步的计算更简单，所以总体准确率更高。

### 深度拆解

**CoT效果对比：**

```
无CoT:
  Q: "一个商店有15个苹果，卖了8个，又进了12个，现在有几个？"
  A: "19个" ← 可能直接猜，容易错

有CoT:
  Q: "一个商店有15个苹果，卖了8个，又进了12个，现在有几个？"
  A: "让我一步步算：
      1. 原有15个苹果
      2. 卖了8个：15-8=7个
      3. 又进了12个：7+12=19个
      答案是19个。" ← 每步简单，准确率高
```

**CoT的变体：**

| 变体 | 原理 | 适用场景 |
|:---:|------|------|
| Zero-shot CoT | 加一句"Let's think step by step" | 通用 |
| Few-shot CoT | 给带推理过程的示例 | 格式控制 |
| Self-Consistency | 生成多条推理链投票 | 数学/逻辑 |
| Tree of Thoughts | 树状搜索多条推理路径 | 复杂规划 |
| Auto-CoT | 自动生成CoT示例 | 省人工 |

**局限性：**

| 局限 | 说明 |
|:---:|------|
| 错误传播 | 中间步骤错了，后续全错 |
| 不保证正确 | 推理链看起来合理但结论可能错误 |
| Token消耗 | 生成更多token，延迟和成本增加 |
| 规模门槛 | 小模型（<60B）CoT反而更差 |
| 简单任务无效 | 翻译、摘要等不需要推理的任务CoT无益 |

### 面试话术

> "CoT让模型在给答案前先写推理过程，效果好的本质是'给模型更多token来计算'——把复杂的多步推理拆成多个简单的单步预测，每步难度降低。
>
> CoT在多步推理任务（数学、逻辑）上效果显著，但需要模型规模够大（60B+）。局限性包括：中间步骤错误会传播、不保证推理链正确、增加token消耗、小模型无效。
>
> 改进方向有Self-Consistency（生成多条链投票取多数）和Tree of Thoughts（树状搜索最优路径）。简单任务如翻译摘要不需要CoT，用了反而增加成本。"

## Q18：大模型为什么会出现幻觉？怎么缓解？

### 一句话破题

幻觉是模型「自信地输出错误信息」——根因是模型本质在做概率预测，不是在做事实检索，训练数据中的噪声和知识边界不清晰导致模型分不清「知道」和「不知道」。

### 核心三问

**Q：幻觉的根本原因是什么？**
模型训练目标是预测下一个token（概率生成），不是存储和检索事实。当遇到训练数据中少见或没有的信息时，模型会根据语言模式生成看似合理但事实上错误的内容。

**Q：能完全消除幻觉吗？**
不能。这是生成式模型的固有特性——只要模型在做概率生成，就有可能生成不符合事实的内容。只能缓解，不能根除。

**Q：RAG能解决幻觉吗？**
能大幅缓解但不能完全解决。RAG提供外部知识减少编造，但模型仍可能误读检索结果或在整合时出错。RAG+好的Prompt+事实校验是最有效的组合。

### 类比记忆

> 🎯 **类比：考试时遇到不会的题**
>
> - **幻觉** = 不会但自信地编了个答案 — 考完还觉得自己答对了
> - **RAG** = 开卷考试 — 翻书找答案，但可能翻错页或理解错
> - **"我不知道"** = 交白卷 — 不编但也没答
>
> 模型的问题是它不知道自己不知道——没有"这道题我不会"的自我认知。

### 深度拆解

**幻觉的五大根因：**

| 根因 | 说明 | 示例 |
|:---:|------|------|
| 训练数据噪声 | 互联网数据本身有错误信息 | 训练数据中的错误事实被模型学到 |
| 知识截止 | 训练数据有截止日期 | 问2025年的事，模型用2023年的知识编 |
| 概率生成本质 | 模型选最"像"的而非最"对"的 | 人名+书名组合出看似合理但不存在的书 |
| 长尾知识 | 训练数据中少见的信息 | 冷门历史事件容易编造 |
| 指令冲突 | Prompt要求回答但模型不知道 | 强行回答而非说不知道 |

**缓解策略（分三层）：**

```
训练层:
  ├─ 高质量数据清洗（去噪、去重）
  ├─ RLHF中奖励"说不知道"的行为
  └─ 知识注入（继续预训练+领域数据）

推理层:
  ├─ RAG（检索增强生成）— 给模型外部知识
  ├─ CoT — 让模型一步步推理减少跳步
  ├─ 自我校验 — "请检查你的回答是否准确"
  ├─ 低温度 — 降低随机性
  └─ 置信度输出 — 让模型标注不确定性

应用层:
  ├─ 事实校验 — 用知识库验证关键事实
  ├─ 引用溯源 — 要求模型标注信息来源
  ├─ 人工审核 — 高风险场景人工复核
  └─ 多模型投票 — 多个模型交叉验证
```

### 面试话术

> "幻觉的根因是模型本质在做概率生成而非事实检索——训练目标是预测下一个token，不是存储和检索事实。当遇到训练数据中少见的信息时，模型根据语言模式生成看似合理但错误的内容。
>
> 缓解分三层：训练层做数据清洗和RLHF奖励'说不知道'；推理层用RAG提供外部知识、CoT减少跳步、低温度降低随机性；应用层做事实校验和引用溯源。
>
> RAG是最有效的缓解手段，但不能完全消除——模型仍可能误读检索结果。幻觉是生成式模型的固有特性，只能缓解不能根除。关键是在高风险场景（医疗、法律、金融）加人工审核兜底。"

## Q19：MoE混合专家模型是什么？DeepSeek V3、Qwen为什么用MoE？

### 一句话破题

MoE（Mixture of Experts）让模型有多个「专家网络」，每次只激活其中几个——用稀疏激活实现「参数量大但计算量小」，在相同计算预算下实现更大模型容量。

### 核心三问

**Q：MoE为什么能省计算？**
传统模型每个token要过所有参数。MoE有多个专家（如8个），每次只用top-2，计算量只有Dense模型的2/8=25%，但参数容量是8倍。实现了「大参数量、小计算量」。

**Q：MoE的代价是什么？**
显存占用大（所有专家都要加载到显存）、训练复杂（负载均衡问题）、通信开销大（多卡训练时专家分布在不同卡上）。

**Q：DeepSeek的MoE有什么创新？**
共享专家（Shared Expert）+ 路由专家。共享专家每次必激活处理通用信息，路由专家按需激活处理 specialized 信息。加上细粒度专家分割和负载均衡loss，训练更稳定。

### 类比记忆

> 🎯 **类比：医院科室**
>
> - **Dense模型** = 每个病人都看全科医生 — 一个医生什么都看，水平有限
> - **MoE** = 医院有8个专科医生，按症状挂号看2个 — 医院总共8个医生的知识量，但每个病人只看2个医生，效率高
> - **DeepSeek MoE** = 8个专科医生 + 1个全科医生(共享专家) — 全科医生每次都看(处理通用信息)，再根据症状加看1-2个专科

### 深度拆解

**MoE架构原理：**

```
输入token
  ↓
路由器(Gate): 计算token对每个专家的权重
  → [0.1, 0.05, 0.7, 0.15, 0.0, 0.0, 0.0, 0.0]
  → 选top-2: 专家2(0.7) + 专家3(0.15)
  ↓
加权组合: output = 0.7 × Expert_2(x) + 0.15 × Expert_3(x)
  ↓
（其余6个专家不激活，不计算）
```

**Dense vs MoE对比：**

| 维度 | Dense（如LLaMA-70B） | MoE（如DeepSeek-V3 671B） |
|:---:|:---:|:---:|
| 总参数量 | 70B | 671B |
| 每token激活参数 | 70B（全部） | 37B（~5.5%） |
| 训练计算量 | 1× | ~0.5×（相同计算量更大容量） |
| 推理计算量 | 1× | ~0.5× |
| 显存需求 | 1× | ~9×（所有专家要加载） |
| 效果 | 基线 | 相同计算量下更好 |

**DeepSeek MoE的创新：**

| 创新 | 说明 |
|:---:|------|
| 共享专家 | 1-2个专家每次必激活，处理通用知识 |
| 细粒度专家 | 把大专家拆成更多小专家，路由更灵活 |
| 负载均衡loss | 防止某些专家过载、某些闲置 |
| 无辅助loss负载均衡 | DeepSeek-V3创新，去掉辅助loss直接调bias |

### 面试话术

> "MoE的核心思想是稀疏激活——模型有多个专家网络，每次只激活top-K个。比如DeepSeek-V3有671B参数但每token只激活37B，实现了'大参数量、小计算量'，相同计算预算下效果超过Dense模型。
>
> MoE的代价是显存大（所有专家都要加载）和训练复杂（负载均衡）。DeepSeek的创新是共享专家+细粒度路由专家——共享专家处理通用信息每次必激活，路由专家按需激活。V3还实现了无辅助loss的负载均衡，简化了训练。
>
| Dense vs MoE选型：显存充足追求效果选MoE，显存有限或追求简单选Dense。趋势上大模型越来越多用MoE。"

## Q20：大模型部署有哪些主流方案？vLLM、TGI、llama.cpp、SGLang怎么选？

### 一句话破题

部署框架选型看场景：高并发生产服务用vLLM（PagedAttention吞吐量最高）、边缘设备用llama.cpp（CPU/GPU通吃）、快速原型用TGI（HuggingFace生态好）、复杂推理场景用SGLang（支持前缀共享和约束解码）。

### 类比记忆

> 🎯 **类比：选车**
>
> - **vLLM** = 高铁 — 高吞吐量，批量运输效率最高
> - **TGI** = 公交车 — 稳定可靠，HuggingFace官方运营
> - **llama.cpp** = 自行车 — 什么路都能走（CPU也行），但运力有限
> - **SGLang** = 定制物流车 — 专门优化特定路线（前缀共享、约束解码）

### 深度拆解

**四大框架对比：**

| 框架 | 核心优势 | 劣势 | 适用场景 |
|:---:|------|------|------|
| **vLLM** | PagedAttention，吞吐量最高 | 显存占用大 | 高并发生产环境 |
| **TGI** | HF生态好，部署简单 | 吞吐量低于vLLM | 快速原型、HF生态 |
| **llama.cpp** | CPU可跑，跨平台 | 并发能力弱 | 边缘设备、个人部署 |
| **SGLang** | 前缀共享、约束解码 | 生态较新 | 复杂推理、多轮对话 |

**vLLM的核心技术——PagedAttention：**

```
传统方式: 每个请求预分配连续显存 → 显存碎片严重，利用率30-40%
PagedAttention: 借鉴OS虚拟内存
  - KV Cache切成固定大小Block（16 token一块）
  - 逻辑Block → Block Table → 物理Block
  - 按需分配，消除碎片
  - 显存利用率 90%+
  → 同样显存能服务更多并发请求
```

**选型决策树：**

```
部署在服务器GPU？
  ├─ YES
  │    ├─ 高并发生产环境？→ vLLM（吞吐量之王）
  │    ├─ 快速原型验证？→ TGI（部署简单）
  │    └─ 复杂推理/多轮对话？→ SGLang（前缀共享）
  └─ NO（边缘设备/个人电脑）
       ├─ 有GPU？→ llama.cpp + GPU后端
       └─ 只有CPU？→ llama.cpp + CPU后端 + INT4量化
```

**部署关键指标：**

| 指标 | 说明 | 影响因素 |
|:---:|------|------|
| 吞吐量 | 每秒处理token数 | 批处理大小、PagedAttention |
| 首token延迟 | TTFT | 模型大小、KV Cache命中率 |
| 每token延迟 | TPOT | 批处理大小、GPU算力 |
| 并发数 | 同时服务多少请求 | 显存大小、KV Cache效率 |
| 显存利用率 | 实际使用/总量 | PagedAttention、量化 |

### 面试话术

> "部署框架选型看场景。vLLM是高并发生产环境的首选——PagedAttention把KV Cache切成Block按需分配，显存利用率从40%提升到90%+，同样显存服务更多并发。TGI是HuggingFace官方框架，部署简单但吞吐量低于vLLM。llama.cpp适合边缘设备，CPU也能跑，配合INT4量化消费级笔记本都能用。SGLang支持前缀共享和约束解码，适合复杂推理场景。
>
| 实际项目推荐：生产环境用vLLM+AWQ-INT4量化，个人实验用llama.cpp。关键指标是吞吐量、首token延迟和并发数，这些主要由批处理策略和KV Cache效率决定。"

## Q21：大模型能力评测指标有哪些？

### 一句话破题

大模型评测分三层：基础能力（MMLU/CMMLU考知识广度）、专项能力（HumanEval考代码、GSM8K考数学）、综合能力（MT-Bench/LMSYS Arena考对话质量）——单一榜单不够，要多维度交叉评估。

**主流评测基准：**

| 类别 | 基准 | 考什么 | 说明 |
|:---:|:---:|------|------|
| **综合知识** | MMLU | 57学科多选题 | 英文，最通用 |
| | CMMLU | 中文多学科 | 中文模型必看 |
| | C-Eval | 中文综合 | 国内主流 |
| **数学推理** | GSM8K | 小学数学应用题 | 基础推理 |
| | MATH | 竞赛数学 | 高难度推理 |
| **代码** | HumanEval | Python函数编写 | 基础代码 |
| | MBPP | 多任务编程 | 更全面 |
| | LiveCodeBench | 实时编程竞赛 | 防数据污染 |
| **对话** | MT-Bench | 多轮对话质量 | GPT-4打分 |
| | AlpacaEval | 指令遵循 | 自动评估 |
| **人类偏好** | LMSYS Arena | 人类盲测PK | 最接近真实体验 |
| **安全** | TruthfulQA | 抗幻觉能力 | 事实准确性 |
| | ToxiGen | 毒性检测 | 安全性 |

| 坑 | 说明 | 应对 |
|:---:|------|------|

> "大模型评测分三层。基础能力看MMLU（英文）和CMMLU/C-Eval（中文），考知识广度。专项能力看HumanEval（代码）、GSM8K（数学）、TruthfulQA（抗幻觉）。综合能力看MT-Bench（多轮对话）和LMSYS Arena（人类盲测PK）。

Step 1: 能力需求
  ├─ 需要顶级推理能力？→ GPT-4o / Claude 3.5
  ├─ 中文为主？→ Qwen2.5 / GLM-4 / DeepSeek
  └─ 代码为主？→ Claude 3.5 Sonnet

Step 2: 部署需求
  ├─ 数据不能出境？→ 开源模型私有部署(Qwen/DeepSeek/LLaMA)
  └─ 可以用API？→ 按能力和成本选

Step 3: 成本预算
  ├─ 预算充足？→ GPT-4o / Claude 3.5
  ├─ 成本敏感？→ DeepSeek-V3（性价比之王）
  └─ 需要私有部署？→ Qwen2.5-72B + vLLM

Step 4: 延迟要求
  ├─ 实时对话？→ 小模型(7B-14B)本地部署
  └─ 可等几秒？→ 大模型API
```

**实际项目选型案例：**

```
场景：企业内部知识库问答系统（中文为主、私有部署）

选型过程：
  ① 能力需求：中文理解+RAG能力 → 候选: Qwen2.5、GLM-4、ChatGLM3
  ② 部署需求：数据不能出境 → 必须开源私有部署
  ③ 硬件资源：2×A100 80G → 可跑72B量化版
  ④ 成本考量：一次性部署成本 vs API按量计费

最终选择：Qwen2.5-72B + AWQ-INT4量化 + vLLM部署
原因：
  - 中文能力最强（CMMLU/C-Eval领先）
  - 开源可商用，私有部署合规
  - INT4量化后35GB，单A100可跑
  - vLLM PagedAttention支撑高并发
  - 社区活跃，微调工具链成熟
```

### 面试话术

> "我对比过GPT-4o、Claude 3.5、Qwen2.5、DeepSeek-V3等主流模型。选型核心看四个维度：能力够不够、成本行不行、延迟可接受不可接受、数据隐私要求。
>
> 上一个项目是企业内部知识库问答，中文为主、数据不能出境。最终选了Qwen2.5-72B做AWQ-INT4量化+vLLM部署。原因：中文能力最强（CMMLU领先）、开源可商用、量化后单A100可跑、vLLM支撑高并发。

> 如果是代码审查场景我会选Claude 3.5 Sonnet，成本敏感的推理任务选DeepSeek-V3，需要多模态选GPT-4o。没有最好的模型，只有最适合场景的模型。关键是要在自己的业务数据上做评测，不能只看公开benchmark。"
